# PRD：多语言（i18n）能力（version 0.1.13）

| 项 | 内容 |
| --- | --- |
| 版本 | `0.1.13` |
| 范围 | 建立站点级 i18n 基础架构；首页（含顶栏）中英双语；首页语言选择器；语言偏好持久化与默认策略 |
| 状态 | 需求已确认（待设计） |
| 默认语言 | `en`（英语） |
| 翻译 key 规范 | 英文 key；`page` / `api` 两大分组；页面组按页面拆分管理 |
| 支持语言 | `zh`（简体中文）、`en`（英语） |
| 关联页面 | `/`（首页）、`PunkHomeHeader`；其余页面本期仅接入基础能力，不承诺全量文案迁移 |

## 背景与目标

7AI·CLUB 当前为 Next.js 15 App Router 项目，界面文案以硬编码中文为主（`layout.tsx` 中 `html lang="zh-CN"`，首页 `PunkLanding`、登录 `/login`、聊天 `/chat`、控制台 `/console`、管理后台 `/admin` 等）。随着产品面向更广受众（含英语读者、海外访客），需要在不破坏现有赛博朋克视觉与业务流程的前提下，引入可扩展的多语言方案。

**业务动机：**

1. 首页作为对外门面，需同时服务中文与英语访客，降低理解门槛。
2. 一次性引入标准 i18n 架构，避免后续各页面重复造轮子。
3. 为控制台、聊天等复杂页面的分批迁移预留统一语言状态与文案组织方式。

**本轮目标：**

1. 引入与 App Router 兼容的 i18n 方案，支持 `zh` / `en` 两种 locale。
2. 首页（`PunkLanding` 主体文案 + `PunkHomeHeader` 导航/登录入口 + 页脚合规信息）完整中英切换。
3. 首页顶栏提供语言选择入口，用户可显式切换语言。
4. 语言偏好可跨会话保持；再次访问时按约定策略恢复语言。
5. 为 antd / dayjs 等组件库 locale 与站点语言联动打下基础（本期至少覆盖首页相关子树）。

**成功指标（可验收）：**

| 指标 | 说明 |
| --- | --- |
| 首页双语覆盖率 | 首页用户可见文案（含 metadata 标题/描述、顶栏、主内容、CTA、特性列表、页脚）100% 具备中英 key |
| 切换即时生效 | 在首页切换语言后，当前页文案在 1 次交互内全部更新，无整页错误或未翻译残留 |
| 偏好持久化 | 用户选择语言后刷新页面、关闭再打开（同浏览器），语言偏好按约定策略保持 |
| 架构可扩展 | 新增一个页面的翻译仅需增加 message 文件与组件调用，不需改动语言切换核心逻辑 |
| 视觉一致性 | 语言切换控件与现有 Punk 赛博风格协调，不破坏顶栏布局与可读性 |

## 目标 / 非目标

### 目标（Goals）

- 建立站点级 i18n 基础设施（库选型、目录约定、Provider/中间层、locale 类型约束）。
- 首页完整中英双语与顶栏语言选择器。
- 定义默认语言、检测顺序、持久化介质（cookie / URL / localStorage 组合策略由设计/开发定稿，见开放问题）。
- 根布局 `html lang` 随当前 locale 动态设置（如 `zh-CN` / `en`）。
- 首页 `metadata`（title、description）随 locale 变化。
- 语言切换后，站内跳转至已接入 i18n 的页面应携带或恢复同一 locale（未接入页面行为见非目标）。

### 非目标（Non-goals）

- **本期不做** 聊天 `/chat`、控制台 `/console`、管理后台 `/admin`、登录 `/login` 等页面的全量文案迁移（可在后续迭代按模块推进）。
- **本期不做** API 响应错误消息、服务端推送文案、LLM 生成内容的多语言（属后端/产品内容域，非 UI i18n）。
- **本期不做** 用户配置中心「界面语言」字段与账号级云端同步（可列为后续增强）。
- **本期不做** 第三语言（如日语、韩语）及 RTL 语言支持。
- **本期不做** 基于 IP/Geo 的自动语言推断。
- **本期不做** 翻译管理平台、Crowdin 等外部协作流程接入。
- **本期不改动** 备案号、ICP 链接等合规信息的法定展示要求（备案号本身不翻译；标签文案如「作者邮箱」可翻译）。

## 用户与核心场景

### 用户角色

| 角色 | 描述 |
| --- | --- |
| 访客（未登录） | 从搜索引擎或外链进入首页，希望用英语或中文了解产品 |
| 已登录用户 | 从首页进入对话/控制台，期望语言选择与后续体验一致（本期仅保证偏好已保存） |
| 管理员/开发者 | 后续迭代中为各页面补充翻译 key，不破坏既有中文体验 |

### 核心场景

1. **首次访问**：访客打开 `/`（或带 locale 的等价路径），系统按默认/检测策略展示**英文**或中文首页（**默认 `en`**）。
2. **主动切换**：访客在顶栏选择「English」或「中文」，首页全部文案即时切换为对应语言。
3. **再次访问**：用户隔天再次打开站点，仍看到上次选择的语言（在 cookie/URL 策略生效范围内）。
4. **跨页导航**：用户从首页点击「对话」「控制台」，本期允许目标页仍为中文硬编码，但语言偏好已写入，供后续页面读取（行为须在开放问题中定稿，避免用户困惑）。
5. **合规阅读**：无论中英文，页脚备案号、邮箱链接保持可点击且信息准确。

## 功能范围

### 本期要做（In Scope）

#### A. i18n 基础架构

- 引入成熟 i18n 库（**期望**与 Next.js 15 App Router 兼容；候选如 `next-intl`，**技术选型不在需求层写死**，由设计/开发阶段确认）。
- 约定支持 locale 枚举：`zh`、`en`（可映射 BCP 47：`zh-CN`、`en`）。
- 约定文案资源组织方式：**两大分组 `page` / `api`**（`api` 组用于 API 返回的用户可见 message）；**`page` 组按页面/模块各自独立文件管理**（见下文「翻译 key 组织规范」）。
- 提供在 Server / Client 组件中消费翻译的统一方式（避免各页面自行 `import` 散乱 JSON）。
- 根 `layout` 或 locale 布局层根据当前 locale 设置 `html lang`。

#### B. 首页语言选择器

- **位置**：`PunkHomeHeader` 顶栏右侧操作区（导航链接与登录/用户菜单之间或相邻），具体布局交给设计。
- **形式**：下拉、分段切换（中/EN）、图标+文字等均可，须满足可访问性（键盘可操作、当前语言有明确态）。
- **选项**：仅 `中文` 与 `English`（或 `ZH` / `EN` 缩写，文案由设计定稿）。
- **交互**：选择后立即更新当前首页文案，无需额外确认弹窗。
- **状态**：当前选中语言在控件上有视觉反馈。

#### C. 首页文案国际化

须纳入翻译 key 的范围（含但不限于）：

| 区域 | 示例文案（现网中文） |
| --- | --- |
| 页面 metadata | `7AI·CLUB`、`个人 AI AI实验场` |
| 顶栏导航 | `对话`、`控制台`、`登录`；`UserAvatarMenu` 在首页 variant 下若有可见文案一并纳入 |
| Hero 标签 | `PERSONAL · AI LEARNING`（可保留英文或中英各一套，由设计定稿） |
| 主标题与副标题 | `解构智能`、`DECONSTRUCT · LEARN · BREAK THINGS` |
| 描述段落 | `AI实验场：模型、提示词、管线——…` |
| CTA | `进入对话`、`后台控制` |
| 特性列表 [01]–[04] | 四条特性说明 |
| 页脚 | `SYS://local · learning mode · no warranty`、`作者邮箱：` 标签（邮箱地址与备案号不翻译） |

#### D. 默认语言与持久化策略（需求层期望）

需求层提出以下**期望**，具体组合由设计/开发确认：

| 维度 | 期望 |
| --- | --- |
| 默认语言 | 无历史偏好时，**默认 `en`（英语）**；检测顺序：cookie/URL 优先 → `Accept-Language` → 默认 `en` |
| 持久化 | 用户显式选择后须跨刷新保持；**优先 cookie**（利于 SSR 与中间件），可辅以 URL locale 前缀（如 `/en`）以支持分享与 SEO |
| URL 行为 | 切换语言后 URL 应能反映或携带 locale（具体是否强制前缀路由，待确认） |
| 登录用户 | 本期可将偏好仅存客户端（cookie）；账号级同步为非目标 |

#### E. antd / dayjs locale 联动（基础）

- 首页子树若使用 antd 组件（如未来语言选择器用 `Select`），其 locale 应随站点语言切换。
- 为 `ConsoleShell` / `AdminShell` 中已有的 `zhCN` 硬编码预留读取「当前 locale → antd locale」的扩展点（本期可不改造 Shell，但架构文档须说明接入方式）。

### 本期不做（Out of Scope）

见上文「非目标」；另补充：

- 不翻译数据库内用户生成内容（昵称、对话内容、知识库正文等）。
- 不翻译管理后台表格列头、表单 label 等（留待后续迭代）。
- 不处理邮件模板、验证码短信等多渠道文案。

## i18n 方案约束（需求层）

以下为产品对技术方案的**约束与期望**，实现细节由开发阶段定稿：

1. **框架兼容**：方案须与 Next.js 15 App Router、React 19、TypeScript 协同，不引入与 SSR/RSC 冲突的全客户端方案作为唯一路径。
2. **双端消费**：支持 Server Component（如 `metadata`、静态区块）与 Client Component（如语言选择器、`use client` 顶栏）共用同一套 message 源，避免双份维护。
3. **locale 路由**：倾向支持基于 locale 的路由或中间件检测（`next-intl` 常见模式）；若采用「仅 cookie、URL 不变」须说明对 SEO 与深链的影响。
4. **类型安全**：翻译 key 宜具备 TypeScript 提示或校验，减少漏翻与拼写错误（实现方式不限）。
5. **与现有结构共存**：尽量不大改 `src/app` 下既有路由树；允许通过 `[locale]` 段或 middleware 渐进迁移。
6. **包管理**：在 `package.json` 中新增明确 i18n 依赖（用户原始需求「使用 i18n 方案」）；具体包名不在 PRD 写死。
7. **key 语言与分组**：翻译 **key 名一律使用英文**；资源按 **`page` / `api` 两大分组** 组织；`page` 组内**按页面各自独立管理**（见下节）。

## 翻译 key 组织规范（已确认）

### 两大分组

| 分组 | 用途 | 示例 |
| --- | --- | --- |
| **`page`** | 页面 UI 可见文案（组件、metadata、顶栏、页脚标签等） | 首页 Hero、导航、CTA |
| **`api`** | API 返回的用户可见 message（错误提示、业务提示等） | `403` 统一错误文案、表单校验 message |

> 注：`api` 组指**前端 i18n 资源中与 API 响应对应的文案**，便于与 UI 文案分离；**本期不改造后端 API 多语言**，但须预留目录与命名约定，供后续迭代接入。

### `page` 组：按页面统一管理

- 每个页面（或页面模块）**单独一份** message 文件，避免单文件膨胀。
- 本期至少包含：`page/home`（对应 `PunkLanding` + `PunkHomeHeader` + 首页 metadata）。
- 后续迭代按页面增量添加，如 `page/login`、`page/chat` 等，**不合并到全局大 JSON**。

**目录示例（locale 为外层，分组为中层）：**

```
messages/
  en/
    page/
      home.json
    api/
      message.json
  zh/
    page/
      home.json
    api/
      message.json
```

### key 命名

- **key 名使用英文**（camelCase 或点分层级，如 `hero.title`、`nav.chat`）。
- **英文 locale（`en`）为 key 的语义源**：`en` 文件中 key 对应英文展示文案；`zh` 文件提供中文翻译。
- 同一 key 在 `en` / `zh` 下结构一致，便于类型校验与漏翻检测。

### 缺失 key 回退

- **生产环境**：缺失翻译时回退至 **`en` 文案**（与默认语言一致）。
- **开发环境**：console 告警 + 可见提示（key 名或占位），便于排查。

## 非功能需求

| 类别 | 要求 |
| --- | --- |
| **SEO** | 各 locale 下首页 `title`、`description` 正确；若采用 locale 前缀 URL，应允许搜索引擎区分中英版本（`hreflang` 可作为后续增强，本期至少 metadata 正确） |
| **可访问性** | 语言选择器支持键盘聚焦与操作；切换后 `html lang` 更新；屏幕阅读器能识别当前语言 |
| **性能** | 语言包按 locale 拆分加载，避免首屏加载全部语言 JSON；切换语言不触发不必要的全站重载（若用软导航更佳） |
| **视觉** | 语言控件融入 Punk 顶栏（cyan/fuchsia 色系、mono 字体可选），移动端不挤压「对话/控制台/登录」主操作 |
| **兼容性** | 不破坏 0.1.12 合规页脚（备案链接、邮箱 mailto） |
| **回退** | 某 key 缺失时，开发环境应有明显告警；**生产环境回退至 `en` 文案**（与默认语言及英文 key 规范一致） |

## 风险与依赖

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| 全站硬编码文案面广 | 本期仅首页双语，用户跨页后语言不一致 | PRD 明确非目标；设计定稿跨页过渡策略；后续迭代按模块迁移 |
| App Router 与 i18n 路由重构 | 可能需调整 `app/` 目录结构 | 首期仅包裹首页与共享 layout；middleware 方案评审 |
| antd Pro 组件文案 | 控制台/后台大量表格内置中文 | 本期不改 Shell；架构预留 `ConfigProvider locale` 动态注入 |
| 双语 slogan 风格 | 标题「解构智能」直译可能失去 Punk 语感 | 设计阶段提供意译方案，允许中英使用不同 slogan |
| SEO 双版本重复内容 | 若 URL 不变仅 cookie，爬虫可能只见一种语言 | 倾向 locale URL 或明确 canonical 策略（开放问题） |
| 测试账户/合规文案 | 0.1.12 登录页测试账号说明等仍为中文 | 不在本期范围，记录为后续迭代 |

**依赖与假设：**

- 假设本期无后端 API 变更；语言偏好纯前端持久化即可。
- 假设英语文案由设计/产品在本迭代提供终稿（无需专业翻译外包流程）。
- 假设**默认展示语言为英语**；中文为对等翻译版本。翻译 key 以英文命名，`en` 为语义源 locale。

## 验收要点（汇总）

1. 项目已引入 i18n 依赖与基础目录，README 或迭代文档中有 locale 扩展说明。
2. 首页在 `zh` 与 `en` 下文案完整、无硬编码中文/英文残留（合规数字与邮箱除外）。
3. 顶栏语言选择器可见、可操作，切换后首页即时更新。
4. 刷新页面后语言偏好保持（按最终定稿的 cookie/URL 策略验证）。
5. `html lang` 与当前 locale 一致；首页 metadata 随语言变化。
6. 备案号链接、邮箱 mailto 在两种语言下均可点击且目标正确。
7. 现有登录、对话、控制台主流程不因本期改动而回归失败（冒烟）。
