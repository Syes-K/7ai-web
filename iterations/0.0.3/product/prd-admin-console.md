| 项 | 内容 |
| --- | --- |
| 版本 | `0.0.3` |
| 类型 | 页面框架 / 导航壳（各业务模块后续迭代填充） |
| 与代码库对齐假设 | 站点已存在 **`/console`**（**普通用户**控制台，源码 `console` 目录，CSR 占位）。**管理后台**与控制台是不同产品概念：管理后台落在 **`/admin/**`（源码 `admin` 目录）**，供**系统管理员**使用；本迭代实现该 admin 路由树下的 **ProLayout 壳子**（此前工程可能尚无完整 `/admin` 树，以实现为准）。 |

### 产品与路由定位（与控制台区分）

| 概念 | 路由 / 源码 | 受众 | 说明 |
| --- | --- | --- | --- |
| **管理后台** | **`/admin/**`**，`admin` 目录 | **系统管理员** | 系统级配置、用户、日志等管理能力；本迭代仅交付壳与一级占位。 |
| **控制台** | **`/console/**`**，`console` 目录 | **普通用户** | 个人/业务侧控制台；**不是**系统管理后台。 |

二者**并行存在**、**职责不同**，不得在文案或入口上混为一谈；本 PRD **仅**覆盖 **管理后台（`admin`）**。

## 1. 需求摘要

### 1.1 背景

产品需要一处统一的**管理后台**入口与布局，供**系统管理员**使用，用于承载运营与系统管理类能力。当前工程已有**普通用户**的**控制台**（`/console`），但缺少**管理后台**侧的标准 **ProLayout** 型框架（左侧导航 + 顶栏）。本迭代仅交付 **`/admin` 下**可扩展的**页面壳子**与一级模块占位，避免后续各功能重复造布局与导航。

### 1.2 目标用户

- **主要**：内部运营、**系统管理员**（通过 **`/admin`** 管理后台访问；与 **`/console`** 普通用户控制台区分）。本版本**不**展开细粒度 RBAC/菜单权限矩阵，权限细则可后续专版需求；**产品与路由上**已明确 **admin（系统管理）** 与 **console（用户控制台）** 两条入口。

### 1.3 核心场景

1. **系统管理员**从 **`/admin`** 进入**管理后台**，看到统一的深色（赛博/科技黑）后台视觉，与站点主风格一致（与 **`/console`** 用户控制台区分）。
2. 用户通过**左侧一级菜单**在下列模块之间切换（**仅占位**，无实质业务表单或列表）：配置管理、用户管理、提示词管理、日志查询、系统助手管理。
3. 用户在**顶部 Header** 区域获得一致的全局上下文（如产品名/后台标题、账户或操作区——细节由设计定；本迭代以**区域存在与布局稳定**为底线）。

### 1.4 成功指标（本迭代可验收）

| 指标 | 说明 |
| --- | --- |
| 框架完整性 | 管理后台具备可用的 **ProLayout** 级布局：**左侧菜单 + 上方 Header**，子路由切换时布局不闪断、不重复挂载整站外壳冲突（与现有根 `layout` 关系由设计/实现协同）。 |
| 导航覆盖 | 五个一级模块均在导航中有**可点击入口**，并落到对应占位页或占位视图（可为空白态 + 模块标题）。 |
| 视觉一致 | 采用 **黑色风格**，与站点现有赛博/科技黑基调一致；具体 token 与对比度由设计输出。 |
| 边界清晰 | 不包含各模块业务 CRUD、复杂筛选、权限矩阵；这些明确列为**不做**或**后续迭代**。 |

## 2. 功能范围

### 2.1 做（In Scope）

- **布局与导航壳**：基于 **ProLayout**（或产品与设计确认的同族能力）实现后台主框架。
- **结构**：**左侧菜单** + **上方 Header**；内容区为子路由出口（占位即可）。
- **一级菜单 / 模块（占位名称）**：
  1. 配置管理  
  2. 用户管理  
  3. 提示词管理  
  4. 日志查询  
  5. 系统助手管理  
- **视觉方向**：**黑色主题**，与全站科技黑一致；菜单选中、悬停、侧栏宽度等由设计定义。
- **路由与目录约定**：管理后台挂载在 **`/admin/**`**（源码 **`admin` 目录**）；**控制台**为 **`/console/**`**（**`console` 目录**），二者用户群与职责不同，与 C 端页、对话页的边界由 layout 与信息架构保证；实现上须能**独立配置**该段的 layout 与 Ant Design Registry 范围，与工程现有约定兼容。

### 2.2 不做（Out of Scope）

- 上述五个模块的**具体业务功能**（增删改查、导入导出、审批流等）。
- **RBAC / 菜单级权限**、审计策略的完整设计（延续项目：角色细分待专版）。
- **后端 API** 的新增或变更（除非为已有登录会话、静态资源加载所必需；本迭代以**前端壳 + 占位**为主叙述）。
- **将管理后台与控制台合并为同一入口**，或**用 `/console` 承担系统管理职责**——本需求**明确不做**；二者保持独立产品与路由（见上文「产品与路由定位」）。

### 2.3 依赖与假设

| 类型 | 内容 |
| --- | --- |
| 依赖 | 用户能使用**既有登录会话**访问需登录的后台类路由（与 0.0.2 会话要求一致）；若当前工程对 `/console` 已有保护策略，**`/admin`** 宜**同等或更严**（系统管理面；由实现评估，本需求要求**未登录不可当作已授权管理员**）。 |
| 假设 | 实现侧将沿用工程对 **CSR + antd / Pro Components** 的方向（见仓库前端 agent 约定）实现 **`/admin`** 管理壳，以避免 App Router 下样式问题；**具体组件版本与 Registry 挂载层级由设计/开发阶段确定**。 |
| 风险 | 若文案或入口将「用户控制台」与「系统管理后台」混称，可能产生心智混淆——须在导航、页面标题与路由上**显式区分** **`/console`** 与 **`/admin`**（本迭代占位页亦应体现模块归属为管理后台）。 |

## 3. 用户故事与验收标准（AC）

### US-ADM-001 进入管理后台并看到统一布局

**作为** 已登录的管理相关人员  
**我想要** 打开管理后台并看到带左侧菜单与顶栏的统一布局  
**以便** 在统一框架下使用后续上线的各模块  

**验收标准：**

- [ ] AC1：进入管理后台根路径或默认子路径时，页面呈现 **ProLayout 风格**的**左侧菜单 + 顶部 Header + 内容区**，无布局错位或明显样式闪断（在约定浏览器环境下）。
- [ ] AC2：内容区为子路由渲染区域；切换一级菜单对应路由时，**侧栏与 Header 保持**，仅内容区变化。

### US-ADM-002 一级模块占位导航

**作为** 管理后台用户  
**我想要** 在左侧看到五个一级模块并可切换  
**以便** 预知后续功能分区并验证导航信息架构  

**验收标准：**

- [ ] AC1：左侧菜单包含且仅包含本 PRD 列出的五个一级项（文案允许设计微调，但语义一致）：配置管理、用户管理、提示词管理、日志查询、系统助手管理。
- [ ] AC2：点击任一一级项可导航至对应**占位页面**（或等价视图），并显示可辨识的**模块标题或说明**（可为简短空状态文案）。
- [ ] AC3：当前选中模块在菜单上有**可见选中态**。

### US-ADM-003 黑色风格与品牌一致

**作为** 产品/品牌方  
**我想要** 管理后台采用**黑色科技风**并与站点现有赛博黑一致  
**以便** 全站体验统一、不割裂  

**验收标准：**

- [ ] AC1：整体为**深色主背景**，关键文本与交互对比度满足可读性（具体对比度由设计给标准，开发落地）。
- [ ] AC2：不出现与全站主品牌明显冲突的亮色主题（除非设计明确作为强调色局部使用）。

### US-ADM-004 未授权访问行为（产品级底线）

**作为** 系统  
**我想要** 未登录用户不能将管理后台当作已授权界面使用  
**以便** 降低误暴露风险  

**验收标准：**

- [ ] AC1：在**无有效会话**时访问管理后台受保护路径，用户被**引导至登录流程**或等价处理，且**不展示**仿佛已登录的管理数据（占位页亦同）。

## 4. 待设计项清单（交接 design）

以下项需 **设计阶段** 产出规格（本需求文档不规定像素级实现）：

| 序号 | 类别 | 说明 |
| --- | --- | --- |
| D1 | 路由与信息架构 | 管理后台根路径为 **`/admin`**（**`admin` 目录**）；与 **`/console`（普通用户控制台）**、首页、对话的**关系说明**（入口来源、是否允许从 console 链到 admin 等由设计/安全策略定，但二者**不得**合并为同一壳）。 |
| D2 | 页面与路由结构 | 五个一级模块对应的**路径片段**、默认落地页、是否需要**重定向**到首个模块。 |
| D3 | ProLayout / 导航 | 侧栏**折叠**行为、菜单图标（占位或线性图标库）、**选中/悬停**状态；是否需 **PageContainer** 标题区。 |
| D4 | Header | 左侧/中间：**后台标题或 Logo**；右侧：**用户菜单、退出**等（与 0.0.2 登出能力衔接）；是否展示环境标签（staging 等）待定。 |
| D5 | 主题与 token | **黑色风格**下的背景层级、分割线、主文字/次要文字、主按钮与链接色；与站点全局 CSS 变量的**映射或扩展**列表。 |
| D6 | 空状态与占位 | 各模块占位页的**空状态插画/文案**、是否统一「开发中」提示组件。 |
| D7 | 响应式 | 最窄支持宽度；窄屏下侧栏是**抽屉**还是**折叠**（产品倾向：至少不破坏可用性）。 |
| D8 | 无障碍与键盘 | 主导航焦点顺序、跳过链接等（可与全站标准对齐，标出最低要求）。 |
