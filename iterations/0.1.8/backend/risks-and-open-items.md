# 风险与待确认项：Turn/Step 文档先行（v0.1.8 / 阶段 3A）

## 1. 主要风险

## R1. 双写一致性风险（Turn + Message）

- 描述：同一轮需要同时写 `Message` 与 `Turn`，失败重试会增加状态漂移概率。
- 影响：可能出现“assistant 已返回但 turn 未终态”或反向不一致。
- 缓解：幂等 `turnId`、重试队列、每日一致性巡检与告警。

## R2. 状态机漂移风险（多端各自推导）

- 描述：若前端与后端同时进行主子阶段聚合，规则不一致会造成 UI 抖动。
- 影响：用户看到状态回退或终态冲突。
- 缓解：以服务端聚合结果为准，前端仅做展示与最小纠偏。

## R3. SSE 事件乱序/丢失风险

- 描述：网络抖动下可能出现增量事件丢失或乱序到达。
- 影响：步骤卡在 `running` 或错误显示。
- 缓解：`seq` 单调号 + 终态快照覆盖 + 非流式/后续拉取补偿。

## R4. Reasoning 信息泄露风险

- 描述：若字段白名单失效，可能输出内部推理或敏感数据。
- 影响：合规与安全风险高。
- 缓解：服务端白名单序列化、统一脱敏中间层、测试用例覆盖敏感样本。

## R5. 存储与性能风险

- 描述：`steps` 记录过细会带来写放大与查询成本上升。
- 影响：高并发下延迟增加，数据库体积快速膨胀。
- 缓解：生产默认存快照优先，事件明细按需/采样持久化。

## 2. 前置依赖

- 已有路由与 pipeline 稳定（参考 `0.1.7` 迁移成果）。
- 需要统一枚举常量（状态、中断原因、主子阶段 key）并对齐前端。
- 需要前端具备“未知字段忽略 + turn 缺失降级”的容错能力。
- 需要日志与指标平台承接 turnId 维度的监控埋点。

## 3. 仍需拍板项（进入 3B 前）

1. **Turn 与 Message 关联方式**  
   采用 `message.turnId` 直连，还是独立映射表 `chat_turn_messages`。
2. **Step 明细持久化粒度**  
   仅保存 `stepsSnapshotJson`，还是额外持久化 `ChatTurnStepEvent`。
3. **灰度切流策略**  
   按租户灰度、按用户灰度，或按会话创建时间灰度。
4. **历史回填策略**  
   仅惰性回填，还是增加离线批处理补齐窗口。
5. **Reasoning safeSummary 长度上限**  
   建议 120/200 字二选一，需统一产品与安全口径。

## 4. 可立即执行的默认决策（若无额外要求）

- 默认采用“`message.turnId` 直连”以减少联表复杂度。
- 默认“快照优先 + 明细可选”以控制存储成本。
- 默认“先小流量租户灰度，再全量”。
- 默认“惰性回填优先，离线回填后置”。
- 默认 `safeSummary <= 200` 字并执行关键字掩码。
