# PRD：对话 Step 状态同步、Reasoning 可见化与 Turn 记录模型

**迭代版本**：`0.1.8`  
**阶段**：全流程阶段 1（需求）  
**关联基线**：`iterations/0.1.7/product/prd-chat-turn-pipeline.md`、`iterations/0.1.7/design/spec-chat-turn-pipeline.md`、`iterations/0.1.7/design/pipeline-branch-steps.md`、`iterations/0.1.7/backend/implementation-notes.md`

## 1. 背景与问题

当前对话窗口仅展示助手最终回复（或流式文本增量），用户无法感知本轮执行到了哪个步骤，也无法区分卡顿发生在前置准备、主模型生成还是后处理阶段。  
`0.1.7` 已完成单轮 pipeline 文档化和服务端最小重构，但“前端阶段状态同步协议（stage telemetry）”明确尚未落地。  
本迭代需要在不破坏既有行为前提下，让用户在前端实时看到 step 状态及基础信息；若存在 reasoning 阶段，也应按产品策略可控展示。  
同时，当前一次问答在数据层主要表现为两条 `message`（用户 + 助手），中间步骤过程缺少统一可追溯记录，需要纳入“单次问答（turn）记录模型”以承载步骤轨迹与快照。

## 2. 目标与成功指标

### 2.1 目标

- 在单轮对话执行期间，前端可实时展示“步骤状态（开始/进行中/完成/失败/跳过/中断）”。
- 每个步骤提供最小必要信息（名称、阶段、时间、状态、可选摘要说明），帮助用户理解系统进度与等待原因。
- 若本轮存在 reasoning 相关阶段，前端默认按 `Level 0` 展示；可展开但受边界约束。
- 与 `0.1.7` pipeline A→F 阶段保持可演进衔接，默认不改变模型输入、摘要写库和消息持久化语义。
- 在数据层引入“单次问答 turn 记录”能力，沉淀每轮步骤过程并对现有 message 模型保持兼容。

### 2.2 成功指标（产品口径）

- 用户在流式会话中可看到至少一次有效 step 状态更新，且终态清晰（完成/失败/中断）。
- 当模型调用失败时，用户仍能看到失败归属步骤和可理解提示，而不只是泛化错误。
- 系统关闭 reasoning 展示或用户无权限时，不泄露中间推理正文，且主对话结果不受影响。

## 3. 用户与核心场景

### 3.1 目标用户

- 终端用户：关注“现在系统在做什么、还要等多久、失败在哪里”。
- 调试/运营角色：关注“哪个环节耗时高、失败率高、是否触发摘要或知识库”。

### 3.2 核心场景

- **场景 A（普通提问）**：用户发送消息后，看到从“已接收”到“生成完成”的阶段推进。
- **场景 B（知识库增强）**：用户看到检索相关步骤状态，理解为何响应有额外等待。
- **场景 C（长上下文）**：触发摘要时，用户看到摘要相关步骤状态，而非误判系统卡死。
- **场景 D（异常中断）**：模型报错、网络中断、用户取消时，前端显示中断态并保留已完成步骤。

- 不要求展示完整 chain-of-thought 原文。
- 不扩展具体 tools/MCP/skills 业务能力（仅保留步骤位置信号）。
- 不定义最终 UI 视觉稿细节（归设计阶段输出）。

## 5. 产品定义：Step 状态同步展示

### 5.1 Step 与状态模型（产品层）

- **Step**：对齐 `0.1.7` A→F 主干，可在前端分组展示（前置、准备、执行、后处理）。
- **状态枚举**：
  - `pending`：已知将执行但尚未开始
  - `running`：执行中
  - `completed`：成功完成
  - `failed`：失败并终止主流程
  - `skipped`：按分支逻辑跳过（例如未触发摘要）
  - `interrupted`：因取消/断流/上游终止导致中断

### 5.2 Step 基础信息（最小可展示字段）

- `stepKey`：稳定标识（便于埋点与前后端映射）
- `stepName`：用户可读名称
- `stageGroup`：A/B/C/D/E/F 分组
- `status`：当前状态
- `startedAt` / `endedAt` / `durationMs`：时序信息
- `message`（可选）：非敏感说明（如“已完成知识库检索”）
- `error`（失败时）：用户可读短描述 + 机器错误码

### 5.3 与现有行为衔接原则

- 默认保留现有 `meta`、`user_message`、`assistant_delta`、`assistant_done`、`error` 事件语义。
- step 同步为增量能力，可通过新增事件类型或扩展 `meta` 字段演进，确保旧前端最小可退化。
- 不改变“用户消息先落库、助手消息后落库、摘要条件触发后落库”的既有顺序。

## 6. 产品定义：Reasoning 展示与权限边界

### 6.1 Reasoning 定义（本 PRD）

- **reasoning 阶段**指主对话过程中可识别的“模型内部推理相关执行态”或“系统生成的推理摘要态”，不等价于完整思维链文本。
- 本期产品要求展示“是否发生 + 当前状态 + 安全级别可展示信息”，不要求原始推理全文外显。

### 6.2 可见性分级

- **Level 0（默认公开）**：仅展示“reasoning 已发生/进行中/完成/失败”等状态，不含内容。
- **Level 1（受控可见）**：展示结构化摘要（例如步骤说明、耗时、结论标签），不得包含敏感提示词或原始推理链。
- **Level 2（受限内部）**：仅在明确授权角色下可查看更详细调试信息；默认关闭。

### 6.2.1 “Level 0 可展开”边界（强约束）

- 展开后**仅允许**显示：
  - 主阶段/子阶段状态时间线
  - `durationMs`、时间戳、终态
  - 原因标签（如中断原因分类）
  - 安全短摘要（可选，长度受限、模板化）
- 展开后**禁止**显示：
  - 原始 chain-of-thought 文本
  - 系统提示词、内部工具调用入参原文
  - 用户隐私正文、密钥、token、连接串等敏感信息
  - 可逆推出内部策略细节的高敏调试字段

### 6.3 权限与边界

- 前端按用户身份/组织策略决定可见等级，未授权时强制降级到 Level 0。
- 任何等级下均不得泄露系统提示、密钥、隐私数据或完整 chain-of-thought。
- 权限校验失败时回退到“不可见 reasoning 内容但保留步骤状态”。

## 7. 关键流程（用户视角）

1. 用户发送消息后，立即看到本轮初始化状态（会话已接收/用户消息已记录）。
2. 前端按事件流更新各 step 状态，展示当前运行步骤与已完成步骤。
3. 若出现 reasoning 阶段，根据权限显示对应可见级别信息。
4. 若某步骤失败，前端标记失败步骤并给出可读错误；后续步骤按规则标记 skipped/interrupted。
5. 本轮结束时，前端显示终态（assistant 完成、失败或中断），并可回看步骤时间线。

## 8. 功能需求（FR）

- **FR-01**：系统应在单轮对话生命周期内同步 step 状态变化到前端。
- **FR-02**：系统应支持步骤分组展示，且至少覆盖 A→F 主干关键节点。
- **FR-03**：系统应支持分支态标记（如摘要未触发时 `skipped`）。
- **FR-04**：系统应在失败时提供步骤级错误归因（步骤标识 + 错误码 + 用户提示）。
- **FR-05**：系统应支持 reasoning 阶段状态展示，并按权限控制可见内容等级。
- **FR-06**：系统应在中断时输出标准化原因分类：`user_cancelled` / `network_disconnected` / `server_timeout` / `unknown`。
- **FR-07**：无论是否开启步骤展示，都不应影响最终助手回复与消息持久化正确性。
- **FR-08**：前端步骤展示采用“主阶段 + 子阶段”双层结构，支持主视图总览与子视图展开。
- **FR-09**：非流式 JSON 响应返回完整步骤快照（整轮终态，不依赖 SSE 增量）。
- **FR-10**：系统应支持 `turn` 级记录，单条记录聚合用户消息、AI 最终回复、步骤轨迹与可选摘要字段。
- **FR-11**：`turn` 与 `message` 双模型并存期间，读写行为需可回退并与历史数据兼容。

## 9. 非功能需求（NFR）

- **NFR-01（兼容性）**：对旧客户端保持向后兼容，新增字段/事件不导致旧版本崩溃。
- **NFR-02（性能）**：步骤同步开销可控，不显著增加首 token 时间和总体响应时延。
- **NFR-03（可靠性）**：事件乱序/丢失时前端可进行最小纠偏（依据时间戳与终态事件）。
- **NFR-04（安全）**：reasoning 展示严格遵循最小披露原则，不输出敏感中间推理。
- **NFR-05（可观测性）**：步骤状态与错误需可被日志/指标系统追踪。

## 10. 失败与中断态展示策略

- **失败（failed）**：某步骤发生不可恢复错误并终止主流程；该步骤标记 failed，后续未执行步骤标记 skipped 或 interrupted。
- **中断（interrupted）**：用户主动取消、网络断连、服务端超时或未知原因终止；保留已完成步骤并显式标记中断原因类别。
- **恢复策略**：前端在下一轮请求可重新初始化步骤面板，历史轮次保留最终状态快照，避免状态污染。

### 10.1 中断原因枚举（协议常量）

- `user_cancelled`
- `network_disconnected`
- `server_timeout`
- `unknown`

## 11. 埋点与可观测性需求

### 11.1 事件埋点（最小集合）

- `chat_turn_step_started`
- `chat_turn_step_finished`
- `chat_turn_step_failed`
- `chat_turn_step_skipped`
- `chat_turn_reasoning_visibility_applied`（记录最终可见等级）
- `chat_turn_reasoning_panel_expanded`（记录展开行为，不含正文）
- `chat_turn_interrupted`

### 11.2 指标维度

- `turnId` / `conversationId` / `stepKey` / `status`
- `durationMs`（step 与整轮）
- `errorCode`（失败时）
- `reasoningVisibilityLevel`
- `isStream`（流式/非流式）

### 11.3 日志与排障要求

- 服务端日志可关联步骤状态变化与当前 run 标识。
- 摘要子调用与主模型调用在观测层可区分，避免混淆。
- 在隐私合规前提下记录必要元数据，不记录敏感正文。

### 11.4 埋点脱敏要求（强制）

- 事件级仅允许上报 ID、枚举、布尔、时长、状态码，不上报用户输入原文与模型完整输出。
- `reasoning` 相关埋点仅上报可见等级、展开动作、摘要长度区间、脱敏标签；不上传推理正文。
- 错误信息需拆分为 `errorCode`（可观测）与 `userMessage`（可展示），禁止上报堆栈中的敏感值。
- 日志和埋点均使用同一脱敏策略（关键字掩码 + 长文本截断 + PII 识别剔除）。

## 12. 新增需求：单次问答记录模型（Turn Record）

### 12.1 需求定义（范围）

- 将“用户问题 + AI 最终答复 + 执行过程”作为**一条 turn 记录**进行聚合存储与读取。
- `turn` 至少包含：
  - `turnId`、`conversationId`
  - `userMessageRef`（或快照）
  - `assistantMessageRef`（或快照）
  - `steps[]`（主阶段 + 子阶段，含状态/时序/原因/可选摘要）
  - `finalStatus`（completed/failed/interrupted）
  - `interruptionReason`（如有）
- `steps[]` 支持按策略记录：
  - 仅结果（状态 + 时长）
  - 结果 + 安全摘要（模板化）
  - 结果 + 调试扩展字段（受权限/环境控制）

### 12.2 约束

- 不改变现有 `message` 写入顺序与核心业务语义（先用户后助手）。
- 对历史会话可“无迁移运行”（读时兼容），不得因新模型导致历史会话不可读。
- 流式与非流式共享同一 `turn` 终态语义；非流式须输出完整步骤快照。

### 12.3 推荐数据建模方案

- **推荐**：采用“`turn` 主实体 + `message` 兼容引用”模式。
- 设计原则：
  - `turn` 是问答聚合与步骤追踪主索引；
  - `message` 继续作为聊天渲染与历史兼容层；
  - 新增 `turnId` 外键（或映射表）关联同轮 user/assistant message；
  - 读取优先 `turn`（有则用），回退 `message` 推断（无则兼容旧数据）。
- 价值：
  - 单轮诊断、回放、统计更直接；
  - 不破坏既有消息接口，可渐进迁移；
  - 便于后续将步骤快照与埋点口径统一。

### 12.4 兼容与迁移原则（0.1.7 与当前 message 模型）

- **原则 P1（双写）**：新会话写入时同时写 `message` 与 `turn`；任一写失败需有补偿或重试策略。
- **原则 P2（读优先级）**：读路径优先取 `turn` 聚合结构；若 `turn` 缺失，回退到现有 `message` 结构。
- **原则 P3（历史兼容）**：历史数据不做阻塞式全量回填；采用惰性回填或离线批处理回填 `turn`。
- **原则 P4（协议兼容）**：现有消息接口字段保持可用，新增字段采用可选扩展，不破坏旧客户端。
- **原则 P5（一致性）**：同一轮中 `turn.finalStatus` 与助手消息终态必须可对齐，避免“消息完成但 turn 中断”。

### 12.5 风险

- 双写阶段存在一致性与重试复杂度上升风险。
- `steps[]` 记录粒度过细时可能带来存储和查询负担。
- 历史回填若策略不当可能造成分析口径断层。

### 12.6 验收标准（AC）

- [ ] AC-TURN-01：新产生问答轮次可生成一条 `turn` 记录并关联对应 user/assistant message。
- [ ] AC-TURN-02：非流式请求响应中包含完整步骤快照，字段与 `turn.steps[]` 一致。
- [ ] AC-TURN-03：旧会话（仅 message）可被正常读取展示，不因缺少 `turn` 报错。
- [ ] AC-TURN-04：`interruptionReason` 仅使用四类标准枚举，且前后端口径一致。
- [ ] AC-TURN-05：reasoning 展开后不出现原始推理正文与敏感字段泄露。

## 13. 验收标准（AC）

- [ ] AC-01：流式会话可见步骤状态时间线，至少包含开始、进行中、终态。
- [ ] AC-02：摘要未触发时，相关步骤展示 `skipped`，而非消失或误报失败。
- [ ] AC-03：模型调用失败时，前端可定位失败步骤且显示可读错误信息。

- [ ] AC-05：启用/关闭步骤展示不改变既有消息持久化顺序与响应语义。
- [ ] AC-06：关键步骤埋点与日志可用于统计成功率、耗时与错误归因。
- [ ] AC-07：前端默认主阶段视图可见，展开后可查看子阶段进度与终态。
- [ ] AC-08：非流式 JSON 返回整轮完整步骤快照。

## 14. 依赖、风险与假设

### 14.1 依赖

- 依赖后端定义可稳定消费的 step 事件协议（SSE 或等价通道）。
- 依赖前端设计统一步骤组件与状态视觉规则。
- 依赖权限系统提供 reasoning 可见等级判定输入。

### 14.2 风险

- 事件粒度过细导致前端噪音和性能开销上升。
- reasoning 边界定义不清导致过度暴露或信息不足。
- 新旧协议并存期若缺少版本策略，可能出现兼容问题。

### 14.3 假设

- 0.1.7 A→F pipeline 仍是本期编排基线。
- SSE 是主要实时通道，非流式场景可用最终态补齐。
