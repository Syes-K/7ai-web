# 文案与交互规范：Step 状态、Reasoning、失败中断

**迭代版本**：`0.1.8`

## 1. 目标

统一状态提示语言与交互动作，减少“系统卡住”误判并提高可恢复性。

## 2. 状态文案规范

## 2.1 主阶段通用文案

- `pending`：`等待执行`
- `running`：`执行中`
- `completed`：`已完成`
- `skipped`：`已跳过`
- `failed`：`执行失败`
- `interrupted`：`已中断`

## 2.2 中断原因文案

- `user_cancelled`：`你已停止本轮生成`
- `network_disconnected`：`网络中断，本轮未完整返回`
- `server_timeout`：`服务响应超时，本轮已中断`
- `unknown`：`本轮意外中断，可重试`

## 3. 行为按钮规范

- `重试本轮`
  - 可见条件：`failed/interrupted`
  - 行为：创建新 turn，沿用用户问题内容
- `查看步骤详情`
  - 可见条件：任意非空 `steps[]`
  - 行为：展开 `StepDetailPanel`
- `收起步骤详情`
  - 可见条件：详情已展开
- `复制诊断信息`
  - 可见条件：`failed/interrupted`
  - 内容：`turnId + stepKey + errorCode + interruptionReason`
  - 禁止复制敏感字段

## 4. Reasoning 面板文案

- 折叠态标题：
  - 未触发：`推理阶段：未触发`
  - 进行中：`推理阶段：进行中`
  - 完成：`推理阶段：已完成`
  - 失败/中断：`推理阶段：异常结束`
- 展开态提示：`仅展示安全摘要与状态信息`

## 5. 可访问性与可用性约束

- 状态颜色必须配合文字与图标，不能仅靠颜色区分。
- 状态更新需触发屏幕阅读器可感知的 `aria-live="polite"` 文本。
- 失败/中断按钮键盘可达，焦点顺序与视觉顺序一致。
- 长时间 `running`（如 >8s）显示进度提示，避免静默等待。

## 6. 异常与降级

- 无 `turn` 数据：显示 `当前会话暂无步骤快照`，保留 message 正常阅读。
- `steps[]` 部分缺失：显示 `步骤信息不完整，已降级展示`。
- 权限不足：reasoning 仅显示状态，不显示扩展信息。

## 7. 验收清单（文案/交互）

- [ ] `failed` 与 `interrupted` 文案区分明确。
- [ ] 四类中断原因文案与协议枚举一一对应。
- [ ] 重试动作创建新 turn，不覆盖历史 turn。
- [ ] 复制诊断信息不包含敏感数据。
- [ ] reasoning 展开态仅显示允许字段。
