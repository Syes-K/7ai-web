## 1. 需求摘要

### 1.1 背景

- `0.1.1` 已定义 **助手** 的主数据（**名称**、**提示词**、**图标（emoji 文案）**、**开场白**、**tags**）及 **个人 / 系统（公有）** 类型，但将「助手如何进入真实对话」延后。
- `0.0.6` 已定义 **`/chat`** 的多会话、持久化、历史列表与清空等能力，尚未要求 **会话级绑定某一助手** 或 **按助手身份展示消息**。
- 本迭代在 **`/chat`** 内闭环：**新建会话时可选择助手，也可不选**；不选则进入与 `0.0.6` 一致的 **普通对话**（无会话级助手绑定、无助手开场白与助手身份展示义务）。若用户 **选择了助手**，则 **会话与助手一经绑定不可在中途更换**；**列表与消息区**对这类会话统一展示该助手的 **图标与名称**；**开场白**作为会话首条助手消息的语义来源（含空值时的产品默认）。

### 1.2 目标用户

- **主要**：已登录、使用 **`/chat`** 的终端用户（与 `0.0.6` 一致）。

### 1.3 核心场景

1. 用户发起 **新对话流程** 时，**可以** 选定一个助手再进入会话，也 **可以跳过选择** 直接进入 **普通对话**（无助手绑定）。若已绑定助手，会话创建后 **不可** 在同一会话内切换为其他助手或「改为普通对话」。
2. 进入 **已绑定助手** 的会话时，对话以 **助手首条消息** 开始，内容来自该助手配置的 **开场白**；若开场白为空，则按本文档 **§3.2** 的明确默认处理。**未绑定助手** 的会话 **不** 要求注入助手开场白，行为与既有普通聊天一致。
3. 在消息列表中，**仅当会话已绑定助手时**，**助手侧消息** 显示该绑定助手的 **图标（emoji）** 与 **名称**；**普通对话** 中助手消息沿用既有「模型」等泛化展示（本迭代不强制改为助手身份）。
4. 左侧 **历史会话列表** 中：对 **已绑定助手** 的会话展示 **助手图标 + 名称**；**无绑定** 的会话不展示助手信息；所有会话均展示 **最后活动时间**（见术语）；**不再** 在会话标题下展示第二行副标题（见范围与验收）。

### 1.4 成功指标（可验收）

| 指标 | 说明 |
| --- | --- |
| 可选绑定 | 新会话流程中用户 **可** 选助手再聊天，也 **可** 不选直接进入普通对话；**已绑定** 会话中 **无「中途换助手」** 入口或等价能力。 |
| 开场一致 | **仅已绑定助手** 的会话：有配置开场白时首条助手消息与配置一致；无配置时行为符合 **§3.2** 默认规则且可验收。 |
| 身份可辨 | **已绑定助手** 的会话：消息区与侧栏中用户能识别 **当前助手**（图标 + 名称），助手消息与身份一致；**普通对话** 不要求助手身份展示。 |
| 历史信息够用 | 侧栏可区分会话、可感知 **最后活动时间**；列表项 **仅标题行**（无副标题行）符合产品要求。 |

## 2. 功能范围

### 2.1 本期要做（In Scope）

| # | 能力 | 说明 |
| --- | --- | --- |
| F1 | **会话前可选助手** | 在用户进入 **新建会话** 或等价流程时，**可** 选定助手后再创建/进入会话，也 **可跳过** 以进入 **普通对话**（无助手绑定）；具体入口形态由设计定，见 §6。 |
| F2 | **绑定不可变（在已绑定时）** | 会话一旦与某助手建立绑定关系，**同一会话内** 不允许更换为其他助手、也不允许「解除绑定改回普通对话」（含无 UI 入口与无有效 API）。**从未绑定助手的普通会话** 不适用本条。 |
| F3 | **开场白为首条助手消息** | **仅** 绑定助手的会话在首次展示时，以 **助手第一条消息** 呈现 **开场白** 内容；实现上可为创建会话时写入或首屏拉取时保证存在，须与 **持久化** 一致（刷新后仍在）。**普通对话** 无此要求。 |
| F4 | **空开场白时的默认** | 见 **§3.2**（仅已绑定助手）：采用 **固定一句产品级通用问候** 作为助手首条消息文案，并以该助手 **图标 + 名称** 展示（与有配置开场白时一致）。 |
| F5 | **消息列表中的助手身份** | **仅** 在已绑定助手的会话中，助手侧消息展示绑定助手的 **图标（emoji）** 与 **名称**；用户消息展示规则保持与 `0.0.6` 一致。普通对话中助手消息展示保持既有行为。 |
| F6 | **历史列表：助手信息** | 对 **已绑定助手** 的会话，列表项展示该助手 **图标 + 名称**；**无绑定** 的会话不展示助手信息（名称可与助手管理中的「名称」字段一致）。 |
| F7 | **历史列表：最后活动时间** | 列表项展示该会话的 **最后活动时间**（定义见 §3.3）。 |
| F8 | **历史列表：无副标题行** | 每个会话条目 **仅保留主标题行**，**移除** 标题下方的副标题/第二行（若当前实现有预览、摘要等占一行，本期从产品上 **取消**）。 |

## 3. 用户故事与验收标准

### 3.1 新建会话前可（可选）选择助手；已绑定则不可更换

**作为** 已登录用户  
**我想要** 在开启新对话前 **可以** 选择要使用的助手，也 **可以** 不选而直接普通聊天；若选了助手，则在该会话中固定使用该助手  
**以便** 需要时用固定人设与提示词，不需要时保持与过去一致的普通对话体验  

验收标准：

- [ ] AC1：存在明确流程，使用户在 **进入可发消息的新会话之前** **可以** 完成 **助手选择**，也 **可以跳过**（或明确「不选助手 / 普通对话」）而 **不建立** 会话级助手绑定；取消/返回行为由设计定。
- [ ] AC2：当用户 **选择了助手** 且会话创建成功后，系统记录 **该会话与所选助手** 的绑定关系，且后续请求使用该助手的 **提示词**（与 `0.1.1` 字段语义一致，具体传参由 backend 定）。
- [ ] AC3：当用户 **未选择助手** 时，会话为 **普通对话**，**不** 写入助手绑定；后续模型调用与 `0.0.6` 无助手场景一致（默认模型与系统提示由实现约定）。
- [ ] AC4：在 **已绑定助手** 的会话中，界面 **不提供**「更换助手」「改用其他助手」「改为普通对话」等会改变绑定关系的操作；若存在设置类入口，亦 **不得** 提供换绑（可与「新建对话」区分）。
- [ ] AC5：用户切换到 **另一会话** 时，可加载该会话各自的绑定状态（有助手 / 无助手）；不同会话之间可不同，**不**要求全局唯一助手。

---

### 3.2 以开场白作为首条助手消息（含空值默认）

**作为** 已登录用户  
**我想要** 进入 **已绑定助手** 的会话时，首先看到该助手的欢迎/引导  
**以便** 明确当前对话身份与语气  

**适用范围**：本节仅适用于 **已绑定助手** 的会话；**普通对话** 无助手开场白注入义务。

**产品规则（空开场白）**：

- 若助手 **开场白非空**：首条助手消息内容 **与配置一致**（trim 等规范化由实现约定，须在测试或文档中说明）。
- 若助手 **开场白为空**：**插入一条固定的产品级通用问候**作为首条助手消息（全站统一文案，例如「你好，我是你的助手，需要什么帮助？」—— **最终文案以设计/文案表为准**），展示上仍使用该助手的 **图标 + 名称**，与自定义开场白无视觉层级差异。

验收标准：

- [ ] AC1：用户首次进入 **新建且已绑定助手** 的会话时，消息列表中 **最先出现** 的为 **助手消息**（在用户发送首条消息之前或同时满足产品期望；至少验收：未发用户消息前即可看到该条助手消息）。
- [ ] AC2：当开场白 **非空** 时，首条助手消息文本与助手配置 **一致**；刷新页面或重新从历史进入，该条消息 **仍存在** 且内容一致。
- [ ] AC3：当开场白 **为空** 时，首条助手消息为 **§3.2** 约定的 **固定通用问候**，且以该助手身份展示；行为在测试用例中可重复验证。
- [ ] AC4：**清空当前会话消息**（`0.0.6`）后，若产品要求「会话仍在」，则重新进入时 **是否再次注入** 开场白/默认问候 —— 须在实现阶段二选一并在验收中写明；**最低要求**：清空后用户可继续对话，不出现无绑定或绑定错乱。（**普通对话** 清空后不适用开场白注入规则。）

---

### 3.3 消息列表展示助手图标与名称

**作为** 已登录用户  
**我想要** 在对话里清楚看到每条助手消息来自哪个助手  
**以便** 与「泛化的模型回复」区分，尤其在多会话切换时  

验收标准：

- [ ] AC1：在 **已绑定助手** 的会话中，每条 **助手角色** 消息旁（或约定位置）展示该绑定助手的 **图标（emoji）** 与 **名称**。
- [ ] AC2：展示用的图标、名称与助手管理中当前配置 **一致**（若助手在管理端被改名，已存在会话是否跟随更新由实现约定：**建议** 会话展示「绑定快照」或「实时拉取」二选一，须在 backend/前端说明中写明并验收一种）。
- [ ] AC3：用户消息展示不因本迭代退化；若界面上曾有仅标注「模型」而无助手名的展示，本迭代应对 **助手绑定会话** 升级为 **助手身份** 展示；**普通对话** 可保持既有「模型」等展示。

---

### 3.4 左侧历史列表：助手信息、最后活动时间、无副标题

**作为** 已登录用户  
**我想要** 在历史列表里快速识别会话对应的助手，并知道最近何时有过互动  
**以便** 更快找到目标会话  

验收标准：

- [ ] AC1：对 **已绑定助手** 的会话，列表项展示该助手的 **图标** 与 **名称**（与 §3.3 信息来源一致）。
- [ ] AC2：列表项展示 **最后活动时间**，语义为 **§2.3**「最后活动时间」；时间格式由设计定（如相对时间/绝对时间），但含义对用户可理解。
- [ ] AC3：**移除** 会话标题下方的 **副标题行**（如消息预览、辅助说明等）；每条仅保留 **主标题行** + 设计约定的 **助手标识** 与 **时间** 等，**不得** 再出现第二行纯文案副标题。（若「时间」独占一行是否算副标题，由设计在稿面确认，但 **不得** 再保留 `0.0.6` 式「标题 + 预览摘要」双行结构中的 **预览摘要行**。）
- [ ] AC4：无助手绑定的历史会话（若仍存在）：列表 **不强行** 展示助手图标；展示规则在实现说明中写明，且列表仍符合 AC3 的 **无预览副标题** 要求。

---

### 3.5 安全、鉴权与数据隔离（补充）

**作为** 系统  
**我想要** 用户只能选择其有权限使用的助手，且会话绑定不可被越权篡改  
**以便** 与 `0.1.1` 个人/系统可见性一致  

验收标准：

- [ ] AC1：助手选择列表仅包含当前用户 **可选用** 的助手（含系统助手与个人助手规则与 `0.1.1` 一致）；不可通过接口为会话绑定 **无权访问** 的助手 ID。
- [ ] AC2：会话列表与消息接口仍仅暴露 **当前用户** 数据（与 `0.0.6` 一致）。

## 4. 依赖与假设

| 类型 | 内容 |
| --- | --- |
| 依赖 | `0.1.1` 助手主数据与权限模型；助手 **提示词** 能注入对话链路（既有 `src/server/llm` 等）。 |
| 依赖 | `0.0.6` 会话与消息持久化、历史列表、`/chat` 路由与登录策略。 |
| 假设 | 会话实体可扩展 **可选** 的 **assistantId**（或等价外键）：有值表示已绑定且不可变；**空** 表示普通对话。创建会话 API **允许** 不传助手或传空。 |
| 假设 | **emoji 图标** 展示规则与 `0.1.1` 一致；过长图标字段的截断策略沿用既有校验。 |

## 5. 风险

| 风险 | 说明与缓解 |
| --- | --- |
| 旧会话与空绑定 | 无助手会话为 **合法常态**；需在实现文档中定义列表/详情与模型链路一致；若提供「补选助手」属未来迭代，本期不强制。 |
| 开场白重复 | 清空会话或重复进入时若重复插入首条助手消息，会导致重复气泡 —— 须在实现中 **幂等**（例如仅当消息数为 0 时注入，或会话级标记已展示开场白）。 |
| 助手配置变更 | 改名/改图标后历史会话展示是否同步 —— 提前在 §3.3 AC2 选定策略并测试。 |
| 与「新建对话」入口叠加 | 若存在多个入口（快捷键、深链接），均需支持 **可选助手**（可默认不选即普通对话）或明确例外并在验收中覆盖。 |

## 6. 待设计项清单（供 design，非设计稿）

- **选助手流程（可选）**：新建对话的入口与 **可选** 选助手的页面/弹层/侧栏步骤；**跳过 = 普通对话** 的路径；取消与返回；默认选中规则（若有，含「默认不选」）。
- **不可换绑的传达**：会话内如何让用户理解「助手固定」；错误场景（如复制会话 ID）是否需要提示。
- **消息气泡**：助手 **图标（emoji）** 与 **名称** 在气泡旁/顶部的布局；与仅「模型」场景的差异化。
- **历史列表项**：单行标题 + 助手标识 + **最后活动时间** 的信息架构；**无预览副标题** 下的留白与截断规则。
- **空开场白** 时 **固定通用问候** 的最终文案与语气（是否区分 B 端/C 端若未来扩展）。
- **清空会话后** 是否再次展示开场白/默认问候 —— 与 §3.2 AC4 对齐的交互偏好。
- **加载/异常**：助手列表加载失败、助手被删除或不可用时，会话打开与历史展示的降级方案（文案与按钮）。
