## 1. 信息架构更新（相对 0.0.6）

### 1.1 顶层分区（不变更语义，增强侧栏与主区职责）

| 区域 | 0.0.6 | 0.1.2 增量 |
| --- | --- | --- |
| **历史区（会话列表）** | 标题、时间、可选预览；选中态 | **移除**消息预览/摘要等**第二行副文案**；新增 **最后活动时间**（语义见 PRD §2.3）；对 **已绑定助手** 的会话展示 **emoji 图标 + 助手名称**（与消息区身份一致）。排序仍为「最近活跃在上」。 |
| **主对话区** | 消息时间线、输入区 | **新建会话路径**增加 **可选助手选择**（选助手 → 绑定会话 / 跳过 → 普通对话）；**已绑定**会话内助手消息展示 **图标 + 名称**；**未绑定**会话助手消息保持既有「模型」等泛化展示（本迭代不强制改版）。 |
| **全局/工具栏** | 新建、清空、移动「历史」入口 | 「新建对话」触发流程扩展为 **先选助手再落会话** 或 **跳过**（见 §2）；**不提供**会话内换绑、解绑、改为普通对话入口。 |

### 1.2 主区信息层次（绑定态 vs 普通态）

| 会话类型 | 主区顶部（可选） | 消息列表身份 |
| --- | --- | --- |
| **已绑定助手** | 若有空间可增加 **轻量上下文条**：emoji + 助手名 + 一句「本对话使用该助手」（**非必须**，避免与列表重复；实现可省略） | 每条 **助手消息** 展示绑定助手的 **emoji + 名称**（§4） |
| **普通对话** | 不展示助手身份 | 助手消息沿用既有 **模型/泛化标签**（与 PRD 一致） |

### 1.3 与 0.0.6 的显式对齐点

- **草稿按 `conversationId` 隔离**（`0.0.6` §5.4）：**延续**；新建会话在服务端返回 `conversationId` 前，输入区可禁用或处于「待创建」态（见 §3.2）。
- **清空确认文案与危险操作**：**延续** `0.0.6` §5.3；清空后对 **开场白是否再注入** 见 §2.4 设计建议（与 PRD §3.2 AC4 对齐）。
- **侧栏宽度 / 断点 / Drawer**：**延续** `0.0.6` §1.2。

## 2. 用户流程

### 2.1 进入 `/chat`（已有历史）

1. 首屏加载会话列表 + 默认选中规则（**延续** `0.0.6` §2.1）。
2. 列表项展示 **标题 + 最后活动时间**；已绑定会话额外 **emoji + 助手名**（§5）；**无预览副标题行**。

### 2.2 新建对话：可选助手 vs 跳过（普通对话）

**推荐入口形态（桌面）**：用户点击 **「新建对话」** → 打开 **Modal**（居中，信息聚焦）或 **右侧 Drawer**（与列表同侧时需注意遮罩层级；**默认推荐 Modal**，减少与侧栏竞争）。

| 步骤 | 用户动作 | 系统行为 |
| --- | --- | --- |
| 1 | 打开选助手界面 | 拉取 **可选用助手列表**（权限与 `0.1.1` 一致）；默认 **不预选**任一助手（满足「跳过 = 普通对话」）。 |
| 2a | 点击 **「跳过 / 普通对话」**（次要按钮，与主行动并列清晰） | 创建 **无 `assistantId` 绑定** 的会话 → 进入主区；**不**注入开场白；消息区助手展示为既有泛化模型标签。 |
| 2b | 选中某一助手后点击 **「开始对话」**（主按钮） | 创建会话并 **绑定该助手** → 主区展示 **首条助手消息**（开场白或 §7 默认问候）；列表出现新条目，带 **emoji + 名称**。 |
| 3 | 点击 **取消** / 遮罩关闭 | **不创建**新会话；焦点回到「新建对话」按钮；当前选中会话不变。 |

**移动**：流程一致；选助手界面全屏 Modal 或全屏 Drawer（宽度 100%），底部固定 **跳过** + **开始对话**，避免误触。

**默认选中**：列表内 **无默认高亮助手**（避免用户未注意而绑定）；若未来产品要求「记住上次所选」，需单独迭代并默认仍提供「普通对话」出口。

### 2.3 切换会话（有/无助手）

1. 用户点击侧栏（或 Drawer 内）另一会话。
2. 主区进入加载态（**延续** `0.0.6` §4.5 防串会话）。
3. 根据该会话 **是否绑定助手** 渲染消息区身份：**有绑定** → 助手消息用 emoji+名称；**无绑定** → 泛化模型展示。
4. **不**在会话内提供「换助手」；用户需 **新建对话** 以使用其他助手。

### 2.4 清空当前会话消息（绑定助手场景）

**产品待定项（PRD §3.2 AC4）**：清空后是否再次注入开场白/默认问候。

| 方案 | 体验 | 设计立场 |
| --- | --- | --- |
| **A：清空后不再次注入** | 消息区为空；展示与「无消息」一致的空态短引导（可区分已绑定：如「向 [助手名] 发送消息以继续」若实现顶栏身份）。 | **推荐默认**：符合「清空 = 删掉记录」心智，避免重复长篇开场。 |
| **B：清空后再次注入一条** | 与首次进入类似，用户始终先看到欢迎。 | 可接受；需 **幂等**（仅一条、不可叠加），且文案不宜过长。 |

**建议**：优先 **方案 A**；若 backend/产品选 **B**，界面层与首次进入 **视觉一致**（同一条助手气泡样式）。

## 3. 状态与交互（助手列表与异常）

### 3.1 选助手界面 — 助手列表加载中

- 列表区 **Skeleton** 或 **Spin**；**「开始对话」禁用**；**「跳过」可点**（若产品接受「无列表也可普通对话」）或二者均禁用至首包返回（二选一：**推荐**加载中仅禁用「开始」，**允许跳过**以降低挫败感）。

### 3.2 可选用助手列表为空

- 展示 **空态插图/图标 + 文案**：如「暂无可选助手」。
- **主按钮「开始对话」禁用**；**「跳过 / 普通对话」高亮可用**，保证用户仍可进入普通对话（满足 PRD「可不选」）。
- 可选次要链：「去助手管理创建」→ `/console/assistants`（与 `0.1.1` 控制台路径一致；若站内无权限则隐藏）。

### 3.3 助手列表加载失败

- 列表区 **Alert** + **重试**；**跳过**仍可用（同上）。
- 全局 **Message** 可选，避免与 Alert 重复。

### 3.4 会话已绑定，但助手被删除或不可用

**历史列表**：

- 若接口仍返回 `assistantId` 但详情拉取失败：展示 **占位**（如灰色默认 emoji +「助手不可用」或「已删除」），**不**伪造名称；**最后活动时间**仍显示。
- 若后端将绑定标记为已清除：按 **无绑定** 展示（与 PRD 旧数据策略一致，以后端为准）。

**打开会话 / 消息区**：

- 顶部或首条提示 **Inline Alert（warning）**：「该助手已无法使用，你可以继续对话或新建对话选用其他助手。」
- 助手消息 **头像/名称**：降级为 **泛化标签**（与「普通对话」一致）或 **保留最近已知快照**（二选一由 backend 定）；设计侧要求 **不出现空白裂图**、**不崩溃**。
- **不提供**换绑；主操作 **新建对话**。

### 3.5 发送中 / 流式 / 错误

- **延续** `0.0.6` §4.2、§4.3；绑定与否 **不改变**发送失败重试逻辑。

### 3.6 无选中会话 / 仅打开选助手未确认

- 取消选助手后：**延续**原选中会话；无选中时 **延续** `0.0.6` 占位与新建引导。

## 4. 视觉与布局 — 消息区（绑定会话）

### 4.1 助手消息（已绑定）

- **位置**：每条助手消息 **气泡上方或气泡左侧对齐竖区**展示一行 **元信息**（与 `0.0.6` 气泡结构兼容）：
  - **Emoji**：与 `0.1.1` 一致为 **文案型 emoji 字符串**（可能多个 codepoint）；单行展示，**超出宽度省略**（`max-width` + `ellipsis`），hover **Tooltip** 全量（防截断歧义）。
  - **名称**：紧随 emoji，**medium 字重**，颜色用 `colorText` 或次要标题色；过长 **ellipsis + Tooltip**。
- **顺序**：`[emoji][名称]` 左对齐；与气泡左缘对齐（助手在左的常规对话布局下）。
- **与用户消息对比**：用户消息 **不**因本迭代改变（延续既有）。

### 4.2 普通对话中的助手消息

- **不**展示助手 emoji/名称；延续 **「模型」或等价泛化标签**（实现现状为准）。

### 4.3 首条开场消息

- 与后续助手消息 **同一套**身份行样式；**无**额外「系统」高亮，避免与聊天内容层级冲突。

## 5. 视觉与布局 — 历史列表项

### 5.1 结构（相对 0.0.6 的核心变化）

| 元素 | 说明 |
| --- | --- |
| **主标题** | **延续** `0.0.6` §6.1（新对话 / 首条用户消息截断 / 已清空回退等）。**单行** `ellipsis`。 |
| **副标题行** | **取消**原「预览摘要」第二行（PRD F8）。 |
| **助手标识（条件）** | 仅 **已绑定**：在标题 **同一行左侧或标题前**展示 **emoji + 名称**（小字号或 Tag 化 **Assistant Chip**），与消息区同源。 |
| **最后活动时间** | **同一行右对齐**（推荐），与标题 **同一行**，避免被误解为「第二段副文案」。窄屏可 **标题换行**，时间仍在首行末列或次行右对齐（**仍不得**出现消息预览）。 |

**布局草图（桌面，单行）**：

```text
[🤖 旅行助手]  上海三日行程相关讨论············  昨天 18:20
```

无绑定会话：

```text
新对话······································  14:32
```

### 5.2 时间与格式（最后活动时间）

**语义**：会话内 **最近一条消息** 的发送/落库时间；无消息时用 **创建时间**（以后端约定为准）。

| 条件 | 展示建议 |
| --- | --- |
| **≤ 1 分钟** | `刚刚` |
| **当天** | `HH:mm`（24 小时制，与站点 locale 一致） |
| **昨天** | `昨天 HH:mm` |
| **本周内** | `周x HH:mm` |
| **更早（本年）** | `M月D日` 或 `MM-DD` |
| **跨年** | `YYYY-MM-DD` |

**与 0.0.6 §6.2**：列表从「更新时间」表述调整为 PRD **最后活动时间**；格式规则 **兼容**原表并细化「刚刚」。

### 5.3 截断与密度

- 标题与助手名 **竞合**时：**标题优先占满中间**；助手 chip **max-width** 约 40% 行宽或固定上限字符，**ellipsis**。
- **选中态**：延续 `0.0.6` §5.2（背景 + 可选左侧色条）。

## 6. 文案与拷贝

### 6.1 空开场白时的默认问候（全站固定一句）

PRD §3.2：开场白为空时插入 **固定产品级通用问候**，展示仍为该助手 **emoji + 名称**。

**建议最终文案（简体中文）**：

> **「你好，我是你的助手。需要什么帮助？」**

- **语气**：中性、简短、可泛用于各类助手；不假设具体领域（与 PRD 示例方向一致）。
- **不**在 UI 单独标注「默认开场」；与自定义开场白 **视觉一致**。

### 6.2 选助手界面（建议）

| 位置 | 文案 |
| --- | --- |
| 标题 | `选择助手` 或 `为本次对话选择助手` |
| 说明 | `选择后将在本对话中固定使用该助手，中途不可更换。你也可以跳过，使用普通对话。` |
| 主按钮 | `开始对话` |
| 次要 | `跳过，使用普通对话` |
| 取消 | `取消` |

### 6.3 空列表 / 错误（见 §3）

文案与 §3 表格一致即可；保持动词清晰（重试、跳过）。

## 7. 可访问性

### 7.1 选助手 Modal / Drawer

| 项 | 建议 |
| --- | --- |
| **打开** | 焦点移至 **容器内第一个可聚焦控件**（**推荐**「跳过」或标题后 **关闭按钮」**，避免未读说明就跳入列表）；若列表可搜索，焦点也可在 **搜索框**。 |
| **焦点顺序** | 关闭 → 说明区（只读可跳过）→ 跳过 → 助手列表（每项为 `role="radio"` 或 `button`）→ 开始对话 → 取消。 |
| **键盘** | `↑`/`↓` 在列表项间移动（可选）；`Enter` 确认当前项并等同「开始对话」；**Esc** 关闭（等同取消，**不创建会话**）。 |
| **焦点陷阱** | 与 antd `Modal`/`Drawer` 默认行为一致；关闭后焦点回到 **「新建对话」**。 |

### 7.2 Emoji 与名称

| 场景 | 建议 |
| --- | --- |
| **列表与消息旁 emoji** | 若名称已完整描述身份，emoji 可对屏幕阅读器 **`aria-hidden="true"`**；**可访问名称**以 **助手名称文本**为准（`aria-label` 含「助手 xxx」可选）。 |
| **仅 emoji 无名称（异常）** | 必须提供 **可见文本或 `aria-label`**，避免读屏静音。 |

### 7.3 历史列表

- **延续** `0.0.6` §7：`aria-current`、列表 `aria-label`。
- 每条会话 **完整可读名称** = 标题 +（若有）助手名 + 时间，避免只读一半。

## 9. 开放问题（供 backend / 产品收口，非 API 规格）

1. **绑定展示策略**：会话展示助手名称/emoji 为 **绑定快照** 还是 **实时跟随** `0.1.1` 配置变更？（PRD §3.3 AC2）影响历史列表与消息区一致性。
2. **无消息会话的最后活动时间**：采用 **创建时间** 是否写死一种规则（PRD §2.3）。
3. **清空消息后** 是否再次注入开场白：**方案 A/B**（§2.4）需实现与验收单一致。
4. **助手删除后** 会话元数据：列表与详情字段如何降级（§3.4）。
5. **深链 / 多入口新建**：若存在快捷键、URL 带 `assistantId` 预填，是否与 Modal 流程一致（PRD 风险「新建入口叠加」）。
