# Backend Implementation Plan（3A 文档先行）- version 0.0.1

## 1. 阶段目标

- 当前阶段：3A（仅服务端文档，不写代码）。
- 本迭代目标：支撑“首页/登录页/对话页/控制台”四页面静态占位交付。
- 关键边界：`0.0.1` 不实现任何服务端逻辑、接口、鉴权、模型调用与持久化。

## 2. 与上游对齐结论

- Product/Design 均明确本版本仅做页面静态占位。
- 前端交付不依赖 API，不需要后端联调。
- 服务端在本阶段仅输出后续可执行方案与接口/模型方向。

## 3. 技术路线（后续版本预留）

1. 服务端框架：Next.js Route Handlers / Server Actions（按场景选型）。
2. 模型编排：LangChain（统一接入与链路编排）。
3. 数据持久化：TypeORM + SQLite。
4. 渲染协同约束：
   - 首页/登录页/对话页：SSR 场景需提供服务端数据入口（后续版本）。
   - 控制台：CSR 场景需提供客户端按需拉取接口（后续版本）。

> 注：以上路线仅作为后续规划，不在 `0.0.1` 落地。

## 4. 分阶段实施建议（面向后续版本）

### 4.1 P1：基础服务能力

- 建立统一响应结构（`code`/`message`/`data`）。
- 建立错误码体系与错误中间层。
- 建立健康检查与基础日志能力。

### 4.2 P2：认证与会话

- 登录/登出/会话查询接口。
- 基础限流、防暴力尝试策略。
- 控制台阶段不区分 `admin/user`，统一用户态访问占位能力。
- 管理后台权限模型（角色细分与授权）延后到后续版本单独立项。

### 4.3 P3：对话能力

- 对话会话与消息存储。
- 通过 LangChain 接入模型调用链路。
- 流式输出与异常降级策略。

### 4.4 P4：控制台能力

- 概览数据接口与配置读写接口。
- 审计字段与变更追踪。
- 后续可接入统计快照。

## 5. 风险与缓解

1. 需求越界风险：静态版本被误加逻辑。
   - 缓解：在 API 与模型文档显式标注 `0.0.1 不实现`。
2. 前后端预期不一致：前端误以为有可用接口。
   - 缓解：前端阶段明确“禁止接口对接”并引用本阶段文档。
3. 技术栈偏移风险：后续实现引入非约定方案。
   - 缓解：文档锁定 LangChain + TypeORM + SQLite 作为基线。
