# PRD：多轮对话摘要（会话级存储 + 管理端配置 + 独立摘要 Agent）

**迭代版本**：`0.1.4`  
**状态**：已对齐实现（2026-04-14）；7.2 仍可在后续迭代细化异步策略

## 1. 背景与目标

- 长对话中，模型上下文窗口有限，需要在服务端维护「对更早轮次的压缩摘要」，并与「最近若干条原文」组合后送入模型，减少信息丢失。
- 项目已在 `DEFAULT_PROMPT_CONFIG` 中预留 `contextSummarySystem`、`summarySystemPrefix` 等模板（见 `src/common/constants/defautPromptConfig.ts`），但**摘要的生成链路、会话持久化与可运营配置**需在本迭代闭环。
- **目标**：实现可配置、可审计的多轮对话摘要；摘要**归属会话（Conversation）**；生成与上下文裁剪优先采用 **LangChain 摘要中间件**（`summarizationMiddleware`），并与主对话链路集成；`maxChars` 作为摘要提示词参数由运行时配置注入。

## 2. 范围说明

### 2.1 纳入范围

| 能力 | 说明 |
|------|------|
| 会话级摘要存储 | 在 `Conversation`（或等价「用户侧 AI 对话会话」）上持久化摘要正文及元数据（如更新时间、可选版本/来源标记）。 |
| 摘要生成 | 使用 LangChain `summarizationMiddleware` 触发摘要子调用；提示词来自合并后的 `contextSummarySystem` / `summarySystemPrefix`；**模型与用户当前聊天一致**（复用 `getChatRuntimeModel`，见 7.1）。 |
| 管理端配置 | 在管理后台「配置管理」中**单独一张 Card**，集中配置摘要相关项；**运行时配置**以 JSON 形式存放于项目 **`data/`** 目录（与现有 `data/app.db` 等并存，具体文件名由设计/后端约定）。 |
| 默认值 | 在 **`src/common/constants/`**（或与 `@/common/constants` 导出一致的位置）提供摘要配置的**默认结构与默认值**，范式参考 `defautPromptConfig.ts`（类型安全、可合并覆盖）。 |

### 2.2 暂不强制（可在设计阶段定稿）

- **用户侧会话 API 不向客户端返回摘要字段**（见 7.3）；摘要仅用于服务端注入主对话上下文，聊天 UI 不依赖摘要全文。
- **摘要触发时机**（同步于发消息前 / 异步等）仍为待定，见 7.2；**设计/后端文档**锁定默认行为与可配参数。

### 2.3 非目标（本 PRD 不展开）

- 多会话合并摘要、跨用户共享摘要。
- 将摘要作为独立「消息类型」对用户可见编辑（除非设计阶段明确纳入）。

## 3. 用户与场景

- **终端用户**：在长对话中与模型交互；期望模型仍「记得」较早讨论过的约束与事实（通过服务端摘要 + 最近原文实现）。
- **管理员**：在配置管理中调整摘要开关、长度上限、提示词模板引用或覆盖、触发频率相关参数等，无需改代码发版（在 JSON 可配置范围内）。

## 4. 功能需求

### 4.1 数据与存储

- **会话（Conversation）** 需能关联：
  - 摘要文本（纯文本即可，首期）；
  - 摘要最后更新时间；
  - 若需要排障：可选 `model`/`token` 粗粒度元信息（设计阶段是否落库）。
- 删除会话时，摘要随会话一并清理（与现有消息级联策略一致）。

### 4.2 摘要生成逻辑（行为级）

- 摘要由 **LangChain 摘要中间件**触发；主对话仍通过 `createAgent` 统一入口执行。
- 提示词体系与现有 `contextSummarySystem`、`summarySystemPrefix` 对齐：`maxChars` 仅作为摘要生成提示词参数。
- **失败策略**（需实现可观测）：摘要子调用失败时，可保留旧摘要或不更新；主对话失败策略沿用既有 `MODEL_ERROR` 行为。

### 4.3 管理后台

- 「配置管理」中新增 **Card：对话摘要**，字段可包括（最终列表以设计稿为准）：
  - 总开关；
  - `maxChars`（摘要生成提示词参数）；
  - 触发模式：`tokens` 或 `messages`，以及对应的 `summaryTrigger*` / `summaryKeep*`；
  - 提示词仍在「提示词模版」维护；运行时 JSON 仅承载开关与数值（**data 覆盖 constants 默认值**）。
- 保存时对 JSON 做校验（schema 或 zod），非法配置拒绝保存并提示。

### 4.4 API（仅需求级，细节见后端文档）

- 管理端：读/写摘要相关 JSON 配置（鉴权与现有 admin 一致）。
- 用户侧：在消息发送等服务端链路中读取/更新会话摘要（具体由 3A API 文档定义）。**不要求**在 `GET /api/chat/conversations/:id`（或同类会话详情接口）响应中返回摘要供前端展示（7.3）。

## 5. 约束与依赖

- 与现有 **`data/`** 目录约定一致：配置文件路径可版本控制忽略或提供示例，部署环境以实际 `data/` 为准。
- 与 **`DEFAULT_PROMPT_CONFIG`**：摘要相关 key 已存在时，**合并策略**需明确（避免两处默认值冲突）；推荐：**摘要业务 JSON** 管「开关与数值」；**提示词片段**继续走现有 prompt-config 合并逻辑，或由本迭代统一归并（设计阶段二选一）。

## 6. 验收标准（草案）

- 新建/延续会话在多轮对话后，**会话记录中存在非空摘要**（在触发条件满足时），且主对话请求中能看到摘要被注入（服务端日志或可调试接口验证）。
- 管理员可在配置 Card 中修改 JSON 可配项并持久化到 `data/`，重启后仍生效。
- 摘要生成可通过 LangChain 中间件链路验证；会话表在触发摘要后可写入 `contextSummary`。

## 7. 已确认与待定项

| 编号 | 议题 | 结论 |
|------|------|------|
| 7.1 | 摘要使用的模型 | **与用户当前聊天相同**：解析方式对齐 `getChatRuntimeModel`，不设单独的「摘要专用模型」配置。 |
| 7.2 | 触发时机 | **待定**（同步于每次用户发消息前 / 异步队列等由设计或后端文档收敛，并可由配置项暴露）。 |
| 7.3 | 会话详情 API 是否暴露摘要 | **不需要**：不在 `GET /api/chat/conversations/:id`（或等价接口）中返回摘要字段；摘要仅供服务端注入上下文。 |
