# PRD：知识库管理（首期：仅直接文本输入）

**迭代版本**：`0.1.6`  
**范围提醒**：本期只实现「直接文本输入」方式的知识库内容维护与使用；文件上传（pdf/txt/md）仅作为**预留扩展点**写入需求与约束，不做实现。

## 1. 背景

当前产品的对话能力逐步完善后，用户希望把**稳定、可复用的知识**沉淀到“知识库”，并在聊天时按需检索注入上下文，以提升回答的一致性与可控性（尤其是项目规范、产品说明、FAQ 等）。

本次迭代先做最小闭环：
- 能创建/编辑知识库条目（仅支持直接文本输入）
- 对话所用助手可配置可多选知识库，并在对话中按需检索注入分片
- 具备可配置的检索参数（topK/阈值），并可观测

## 2. 目标 / 非目标

### 2.1 目标（Goals）

- 提供可用的「知识库条目」管理能力：新增、查看、编辑（首期可不含删除，见 11）。
- 知识库内容可向量化并写入现有 sqlite（新增“知识库向量数据表”），供聊天检索使用。
- 对话所用助手支持配置 0~N 个知识库，并在对话中按需检索注入相关分片。
- 提供可配置的检索参数：`topK` 默认 3；`置信度阈值` 默认 0.75；并定义配置优先级与生效方式。
- 为未来文件上传（pdf/txt/md）预留：UI 入口、接口形态、数据结构扩展点，不阻塞后续迭代接入。

### 2.2 非目标（Non-goals）

- 本期**不做**文件上传解析、OCR、格式化提取、异步导入队列。
- 本期**不做**高级知识库能力：分组/空间、多租户共享、权限细分（到条目级）、版本管理、回滚、协作编辑、审核流。
- 本期**不做**复杂检索策略：混合检索（BM25+向量）、重排序、跨库去重、结果引用展示 UI（可在后续迭代补充）。
- 本期**不承诺**完整的可观测体系平台化接入；仅要求满足最小排障日志（见 10）。

## 3. 用户与核心场景

### 3.1 用户角色

- **普通用户（使用者）**：维护自己的知识库内容，并在对话中使用已配置的助手知识库辅助回答。
- **管理员（运营/开发/管理端用户）**：配置全局默认检索参数、查看基础日志/统计（如有管理端配置页面）。

### 3.2 核心场景

1) 用户把“项目背景/产品定义/FAQ”以文本形式录入某个知识库。  
2) 用户开始对话，系统按对话所用助手配置的知识库集合进行意图识别与检索（若触发）。  
3) 对话过程中，系统先基于所选知识库的**名称+描述**进行意图识别：判断是否需要检索知识库。  
4) 若需要，则从所选知识库中检索 topK 分片并按阈值过滤，注入会话上下文，再生成回答。  
5) 用户更新知识库内容后，向量化结果同步更新，后续对话检索生效。

## 4. 功能范围

### 4.1 本期要做（In Scope）

#### A. 知识库条目管理（仅直接文本输入）

- 新增知识库条目
  - 录入：名称、描述、tags、内容格式（md/普通文章，字段名待定）、内容正文（直接文本输入）
  - 保存后触发向量化（见 6）
- 查看知识库条目详情（展示字段与元信息）
- 编辑知识库条目（更新字段后重新向量化）

#### B. 助手管理（控制台）配置与对话使用

- 在**控制台的「助手管理」页面**为“某个助手”配置可用知识库（可多选）
  - 该配置为**助手级**配置（非会话临时选择）
  - 对话时：系统以当前对话所使用的“助手”之配置作为知识库集合
- 每轮对话（或按一定策略）执行：
  - 基于该助手已配置知识库的**名称+描述**做意图识别：判断是否需要检索知识库
  - 需要时从该助手已配置的知识库中检索分片并注入会话上下文

#### C. 分片测试（调试/验收工具）
- 在知识库管理中提供“测试分片/检索”的入口（建议放在知识库详情页或编辑页附近）
- 输入：一段测试文本（作为检索 query）
- 行为：对该知识库执行向量检索/分片召回（不需要意图识别、不需要对话式生成）
- 输出：结果列表中展示命中分片的
  - 顺序（按相似度排序的 rank）
  - `chunkIndex` / 原始分片序号
  - 置信度/相似度分数（用于与阈值同口径的展示）
  - chunk 内容预览（用于人工验证）
- 默认规则：topK 默认 3、置信度阈值默认 0.75；低于阈值的命中不展示，且列表最多展示 topK 条

#### D. 可配置检索参数

- `topK` 与 `置信度阈值` 支持配置，并有默认值（topK=3，阈值=0.75）
- 配置的生效范围与优先级（建议）：
  - **会话/助手级（用户临时选择）** > **用户偏好（如有）** > **全局默认（管理端）** > **代码默认**
  - 若当前产品尚无“用户偏好/管理端”配置入口，本期至少保证**代码默认**与**会话临时值**（若 UI 提供）两级可用；其余作为预留扩展点记录在 5.2/11

### 4.2 预留扩展点（Out of Scope 但需预留）

#### 文件上传（pdf/txt/md）

本期不实现，但需在需求层面明确预留：
- **UI 预留**：在“新增/编辑知识库条目”页预留上传入口位置与交互占位（禁用态或“即将支持”说明），不影响文本输入主路径。
- **接口预留**：知识库内容的输入类型支持枚举扩展（如 `sourceType=text|file`），但本期仅允许 `text`。
- **数据结构预留**：条目可扩展存储“来源文件元信息”（文件名、类型、大小、hash、上传时间、解析状态等），本期可为空。

## 5. 关键流程（用户视角）

### 5.1 创建知识库条目（文本输入）

1. 用户进入知识库管理（入口见 8.2 待设计/产品交接项）  
2. 点击“新建知识库”  
3. 填写名称/描述/tags/内容格式/正文  
4. 点击保存  
5. 系统保存条目并执行向量化  
6. 返回详情页，展示“向量化状态：成功/失败/进行中（若有）”与最近更新时间

### 5.2 在对话中使用知识库（基于助手配置）

1. 管理员/用户进入控制台「助手管理」  
2. 进入某个助手的配置页，在“知识库”区域选择 0~N 个知识库并保存  
3. 用户使用该助手进行对话并发送消息  
4. 系统基于（该助手已配置知识库的名称+描述）与用户消息做意图识别：
   - 若识别为“不需要知识库”，则不检索直接走原对话
   - 若识别为“需要知识库”，则检索分片并注入上下文
5. 模型生成回答并返回

## 6. 向量化需求（实现约束级）

> 本节描述“必须实现的行为与约束”，不包含代码细节。

- **向量化模型**：单独使用向量化 model 创建**单独的 agent**（与主对话 agent 解耦），用于 embedding/向量化过程。
- **存储**：向量结果存入现有 sqlite；新增“知识库向量数据表”（表结构见 9.2）。
- **分片**：
  - 使用 `RecursiveCharacterTextSplitter`
  - 若内容格式为 markdown，分片策略需等价于 `fromLanguage("markdown")`
  - 普通文章格式使用默认分片策略
- **触发时机**（本期默认）：
  - 新增成功后触发一次向量化
  - 编辑保存后触发一次向量化（并清理/覆盖旧向量数据，策略见 9.3/11）
- **失败策略**：
  - 向量化失败不应阻塞条目保存（条目仍可查看/编辑），但需记录失败状态与可排障信息（见 10）

## 7. 检索与注入策略（需求级）

### 7.1 意图识别（先于检索）

- 输入：用户当轮消息 + 所选知识库的（名称、描述）
- 输出：是否需要检索知识库（boolean）以及可选的“检索原因/信号”（用于日志）
- 约束：
  - 未选择任何知识库时，不进行意图识别与检索
  - 当识别结果不确定时，默认“保守检索”或“保守不检索”需在 11 决策；推荐默认：**保守检索**但受阈值过滤约束，降低漏召回风险

### 7.2 检索

- 在所选知识库范围内检索相关分片
- 结果按置信度阈值过滤（默认 0.75）
- 取 topK（默认 3）
- 注入会话：将命中的分片内容作为“参考上下文”加入本轮模型输入（具体拼接模板由设计/后端文档细化）

## 9. 数据与字段说明（信息架构：实体字段与约束）

### 9.1 知识库实体（KnowledgeBase）

> 说明：本期按“一个知识库条目 = 一份可检索内容”定义。若未来要支持“一个知识库包含多文档”，需在 11 另行扩展为 KnowledgeBase + KnowledgeBaseDocument。

| 字段 | 类型 | 必填 | 约束/说明 |
|------|------|------|----------|
| id | string/uuid | 是 | 系统生成 |
| name | string | 是 | 1~64 字；同一用户范围内建议唯一（是否强校验见 11） |
| description | string | 否 | 0~500 字；用于意图识别与选择器展示 |
| tags | string[] | 否 | 0~20 个；单 tag 1~20 字；用于筛选（若有） |
| contentFormat | enum | 是 | `markdown` / `plain`（字段名待定，见 open questions） |
| content | string | 是 | 直接文本输入；最大长度建议先设上限（见 11） |
| sourceType | enum | 是 | 本期固定 `text`；预留 `file` |
| createdAt / updatedAt | datetime | 是 | 系统维护 |
| vectorStatus | enum | 是 | `pending/success/failed`（或等价） |
| vectorUpdatedAt | datetime | 否 | 最近一次成功向量化时间 |
| vectorError | string | 否 | 失败原因摘要（需脱敏） |

### 9.2 知识库向量数据表（KnowledgeBaseVectorChunk）

| 字段 | 类型 | 必填 | 约束/说明 |
|------|------|------|----------|
| id | string/uuid | 是 | 系统生成 |
| knowledgeBaseId | fk | 是 | 关联 `KnowledgeBase.id` |
| chunkIndex | int | 是 | 从 0 开始递增 |
| chunkContent | text | 是 | 分片内容（可做截断存储策略，见 11） |
| chunkMeta | json | 否 | 预留：起止字符、标题层级、来源等 |
| embedding | blob/json | 是 | 向量数据（存储形式由后端方案确定） |
| embeddingModel | string | 是 | 记录使用的向量化模型标识，便于回溯 |
| createdAt | datetime | 是 | 系统维护 |

### 9.3 更新与清理策略（需求）

- 编辑知识库内容后，必须保证检索不混入旧内容。
- 默认推荐策略：
  - 重新分片并写入新向量前，先按 `knowledgeBaseId` 清理旧 chunk（或写入新版本后切换指针），具体实现由后端方案决定
- 删除策略本期可不做，但需明确“未来删除时需级联删除向量数据表记录”（见 11）

## 10. 埋点 / 日志 / 可观测（最小要求）

### 10.1 必须日志（用于排障）

- **向量化任务日志**（按知识库条目粒度）
  - knowledgeBaseId、内容格式、内容长度、分片数、向量化模型、耗时、状态（success/failed）与失败原因摘要
- **检索日志**（按会话/消息粒度）
  - 会话/消息 id、所选 knowledgeBaseIds、意图识别结果（needSearch true/false）与置信度（若有）、topK/阈值、命中 chunk 数、最终注入 chunk 数、耗时
- **异常日志**
  - sqlite 写入失败、向量化模型调用失败、检索失败、配置非法（topK/阈值越界）等

### 10.2 可选埋点（如当前项目已有埋点体系）

- 创建/编辑知识库成功次数
- 启用知识库检索的对话轮次占比
- 检索命中率（命中 chunk 数 > 0 的比例）

### 10.3 分片测试日志（最小要求）
- 当用户发起“测试分片/检索”时记录：
  - knowledgeBaseId、query 文本长度、topK/阈值
  - 命中数量与最终展示数量（展示数量受 topK/阈值影响）
  - top result 的相似度/置信度分数与耗时（或至少最大分数与耗时）

## 11. 边界、异常与待定决策（本期默认建议）

> 完整待定项清单见 `open-questions.md`，此处列出与可验收相关的关键边界。

- **内容长度上限**：建议首期限制 `content` 最大长度（例如 50k 字符）并在超限时阻止保存；默认值待定。
- **名称唯一性**：建议同一用户范围内 name 唯一；若冲突提示“名称已存在”；是否强制需确认。
- **删除能力**：本期可不做删除；若做，需级联删除向量 chunk；默认建议：**不做删除，仅做编辑**，通过“禁用/归档”作为后续扩展。
- **意图识别不确定**：默认建议：不确定时也执行检索，但最终仍受阈值过滤；避免错过知识召回。
- **未向量化/向量化失败**：聊天检索应跳过该知识库或提示“不可用”；默认建议：跳过并记录日志，不阻塞对话。
- **多知识库冲突**：多选时可能产生重复/矛盾片段；首期只要求按相关度取 topK，不做跨库去重/重排序。

## 12. 验收标准（PRD 级摘要）

> 详细可测试条目见 `acceptance-criteria.md`。

- 用户可创建知识库条目并通过“直接文本输入”保存成功；保存后可看到向量化状态最终为成功（在模型可用时）。
- 用户可编辑知识库条目；编辑后检索结果反映新内容，不混入旧内容。
- 对话所用助手可配置 0~N 个知识库；用户发送消息后，系统能按意图识别决定是否检索，并在需要时注入分片。
- `topK` 默认 3、阈值默认 0.75 生效；当配置非法时有明确错误提示/日志且不导致系统崩溃。
- 文件上传入口在 UI/接口/数据结构层面均有明确预留点，但本期不可用且不影响文本输入流程。
- 知识库完成向量化后，提供“测试分片/检索”能力：输入 query 后在结果中展示 rank 顺序、chunkIndex 以及置信度/相似度分数，并遵循 topK 与阈值过滤规则。
