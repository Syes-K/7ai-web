# 验收标准：知识库管理（0.1.6）

> 说明：本文件按功能点给出可测试条目，默认使用 Given/When/Then 结构。  
> 范围约束：本期只支持**直接文本输入**；文件上传（pdf/txt/md）仅验收“预留扩展点是否存在且不可用”。

## A. 知识库条目 - 创建（直接文本输入）

### AC-A1：必填字段校验

- Given 用户进入“新建知识库条目”
- When 用户未填写名称或未填写正文内容并点击保存
- Then 系统阻止保存并提示必填项缺失（提示文案可由设计定稿）

### AC-A2：名称长度与格式约束

- Given 用户进入“新建知识库条目”
- When 用户输入名称长度超出限制（默认 64 字；若最终不同以实现为准）并点击保存
- Then 系统阻止保存并提示名称超限

### AC-A3：tags 数量与单项长度约束

- Given 用户进入“新建知识库条目”
- When 用户添加 tags 数量超过上限（默认 20）或单 tag 超过上限（默认 20 字）并点击保存
- Then 系统阻止保存并提示 tags 约束不满足

### AC-A4：内容格式选择

- Given 用户进入“新建知识库条目”
- When 用户选择内容格式为 markdown 或普通文章（plain）并填写正文后保存
- Then 条目保存成功，且条目上记录的内容格式与选择一致

### AC-A5：创建成功后可查看详情

- Given 用户已成功创建一条知识库条目
- When 用户进入该条目的详情页
- Then 用户能看到名称、描述、tags、内容格式、正文内容以及“向量化状态/更新时间”（字段名可不同，但信息等价）

## B. 知识库条目 - 编辑与更新

### AC-B1：编辑保存后字段更新生效

- Given 用户打开某条知识库条目的编辑页
- When 用户修改名称/描述/tags/正文并保存
- Then 返回详情页时展示为新值

### AC-B2：编辑保存后向量结果与新正文一致（不混入旧内容）

- Given 用户对条目正文做了明显变更并保存
- When 使用配置了该知识库的助手进行对话并发送一个明确依赖新正文的信息提问
- Then 检索注入的内容（通过日志或调试手段观察）应来源于新正文；不应出现旧正文专有片段

## C. 向量化（Embedding）与分片

### AC-C1：创建后触发向量化

- Given 用户成功创建知识库条目
- When 系统完成向量化流程
- Then 条目的向量化状态为 success（或等价状态），且向量数据表存在与该条目关联的分片记录（chunk 数 > 0）

### AC-C2：markdown 分片策略

- Given 用户创建内容格式为 markdown 的知识库条目，正文包含 markdown 结构（如标题/列表/代码块）
- When 系统完成向量化分片
- Then 分片策略与 markdown 语言分片要求一致（等价于 `fromLanguage("markdown")`），且不出现整段合并为单一 chunk 的异常（除非正文极短）

### AC-C3：plain 分片策略

- Given 用户创建内容格式为 plain 的知识库条目，正文为长段落文本
- When 系统完成向量化分片
- Then 分片策略与默认 `RecursiveCharacterTextSplitter` 行为一致，chunk 数合理（>1，除非正文很短）

### AC-C4：向量化失败不阻塞条目保存

- Given 向量化模型不可用或向量化过程发生错误
- When 用户创建或编辑知识库条目并保存
- Then 条目仍保存成功可查看/可再次编辑；条目向量化状态为 failed（或等价），并记录失败原因摘要（需脱敏）

## D. 控制台（助手管理）- 选择知识库（可多选）

### AC-D1：支持 0~N 个知识库选择

- Given 用户进入控制台「助手管理」并打开某个助手的配置页
- When 用户不选择任何知识库并保存助手配置，然后使用该助手进行对话发送消息
- Then 系统不进行知识库检索，回答流程不因知识库模块报错

- Given 用户进入控制台「助手管理」并打开某个助手的配置页
- When 用户为该助手选择 2 个及以上知识库并保存配置，然后使用该助手进行对话发送消息
- Then 系统在该助手已配置知识库范围内执行意图识别与检索（若触发），不因多选导致崩溃

### AC-D2：所选知识库在助手配置内生效

- Given 用户已为某个助手在控制台配置了至少一个知识库
- When 用户使用该助手进行对话并连续发送多条消息（同一会话）
- Then 该助手配置的知识库对后续消息仍生效，直到用户在控制台修改该助手配置（具体交互以设计为准，但需行为一致且可验证）

## E. 意图识别 → 检索 → 注入

### AC-E1：基于名称+描述做意图识别

- Given 用户选择了至少一个知识库，且该知识库具有名称与描述
- When 用户发送一条与该知识库主题明显无关的消息
- Then 意图识别结果为“不需要检索”（needSearch=false）或等价行为（不发起检索）；并记录可排障日志

### AC-E2：需要时检索并注入分片

- Given 用户选择了至少一个知识库，且正文包含某些可唯一定位的信息点
- When 用户发送明确依赖该信息点的问题
- Then 意图识别结果为“需要检索”（needSearch=true），且检索命中分片被注入本轮上下文（可通过日志/调试手段验证）

### AC-E3：topK 默认值生效

- Given 用户未显式修改 topK（无会话临时配置）
- When 触发检索
- Then 最终注入分片数量不超过 3（topK=3），若命中不足 3 则按实际注入

### AC-E4：阈值默认值生效

- Given 用户未显式修改阈值（无会话临时配置）
- When 检索返回包含高/低置信度结果
- Then 系统按阈值 0.75 过滤；低于阈值的结果不应被注入

### AC-E5：配置越界的保护

- Given 系统存在可配置入口（管理端或会话临时配置）
- When 将 topK 配置为非正整数或将阈值配置为 <0 或 >1
- Then 系统拒绝保存该配置（或回退到默认值并提示）；且不会导致对话崩溃

### AC-E6：向量化失败的知识库不参与检索

- Given 用户选择了一个向量化状态为 failed（或无向量数据）的知识库
- When 用户发送可能需要检索的消息
- Then 系统应跳过该知识库的检索（或视为无可用分片）；对话仍可完成，且记录日志说明跳过原因

## F. 文件上传（预留扩展点，仅验收“不可用且不影响主流程”）

### AC-F1：上传入口存在且不可用

- Given 用户进入知识库条目新建/编辑页面
- When 用户查看“文件上传（pdf/txt/md）”相关入口
- Then 入口以禁用态/占位态呈现，且提示“本期不支持/即将支持”，并且不影响直接文本输入保存

### AC-F2：接口/数据结构可扩展（需求验收）

- Given 需求/接口文档中存在内容来源类型（如 `sourceType`）的扩展设计
- When 评审本期实现范围
- Then 明确本期仅允许 `text`，且后续可扩展 `file` 而不破坏既有数据与 UI 主流程

## G. 日志（最小可观测）

### AC-G1：向量化日志齐全

- Given 用户创建或编辑知识库条目触发向量化
- When 向量化成功或失败
- Then 日志中至少包含 knowledgeBaseId、内容格式、内容长度、分片数、向量化模型、耗时、状态与失败原因摘要（失败时）

### AC-G2：检索日志齐全

- Given 用户使用某个已在控制台配置知识库的助手进行对话并发送消息
- When 意图识别与检索流程运行
- Then 日志中至少包含会话/消息标识、该助手配置的 knowledgeBaseIds、needSearch、topK、阈值、命中/注入数量与耗时

## H. 分片测试（用于验证分片与召回）

### AC-H1：测试入口可用（前置：向量化完成）

- Given 用户存在一条知识库条目，并且其向量化状态为 success
- When 用户在该知识库进行“测试分片/检索”，输入一段 query 文本并触发测试
- Then 系统展示测试结果列表（若无命中则列表为空），且不影响正常的知识库查看/编辑流程

### AC-H2：结果按相似度排序并展示顺序

- Given 用户触发一次“测试分片/检索”
- When 系统返回多个命中分片
- Then 结果列表的显示顺序必须与系统检索的排序一致（rank 从 1 开始）

### AC-H3：展示 chunkIndex 与置信度/相似度分数（便于测试）

- Given 用户在“测试分片/检索”中获得命中结果
- When 系统渲染结果列表
- Then 每条结果至少展示：`rank`、`chunkIndex`（或等价原始分片序号）、置信度/相似度分数、chunk 内容预览（可截断）

### AC-H4：topK 与阈值过滤生效

- Given “测试分片/检索”使用默认配置（topK=3、阈值=0.75）
- When query 命中条目超过 topK 或存在低于阈值的结果
- Then 展示结果最多 3 条，且任何低于阈值的结果不得出现在列表中

### AC-H5：编辑后向量化更新生效

- Given 用户已编辑某条知识库内容并完成重新向量化
- When 用户对该知识库使用“测试分片/检索”，输入能依赖新正文信息的 query
- Then 结果列表中命中的 chunk 内容来源应与新正文一致，不应出现旧正文的专有片段（以可观察证据或日志验证）

### AC-H6：向量化失败时跳过或提示

- Given 用户选择了一个向量化状态为 failed（或无向量数据）的知识库
- When 用户发起“测试分片/检索”
- Then 系统应返回空结果或给出明确不可用提示；并记录日志说明跳过原因
