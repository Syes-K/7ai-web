# Requirements - 0.0.2 登录与注册

## 1. 功能需求

### FR-AUTH-001 注册页与注册流程

- 系统必须提供**注册页**，可通过独立路由访问（建议路径 `/register`，实现阶段可与设计对齐）。
- 注册表单必须收集以下字段：
  - **`email`（邮箱）**：**必填**；须通过常见邮箱格式校验（前后端至少各一层，后端为最终裁决）；在系统内**唯一**。
  - **`telNo`（手机号）**：**可选**；若用户填写，则必须为 **11 位数字** 字符串；不接受含空格、横线、国家码前缀等非纯 11 位形式。若填写且在系统内与他人冲突，按「手机号已被占用」处理。
  - **`nickName`（昵称）**：非空展示名；长度与非法字符限制见本文档「字段规则」。
  - **`password`（密码）**：符合本文档「密码策略」。
  - **`passwordConfirm`（确认密码）**：与 `password` 一致方可提交（产品要求：减少误输入；实现可在客户端先验，服务端仍须校验密码字段完整性）。
  - **图形验证码**：**必填**；用户须输入与当前验证码图片（或等价展示）一致的校验信息；错误时可刷新验证码后重试（见 FR-AUTH-009）。
- 提交成功后：用户处于**已登录**状态，并执行 FR-AUTH-006 中的跳转与会话要求。

### FR-AUTH-002 登录页与登录流程

- 系统必须提供**登录页**（与现有 `/login` 衔接），替换 0.0.1「仅占位、不可提交」的定位。
- 登录表单必须包含：
  - **账号**：**邮箱 `email`**，作为唯一登录标识（不使用手机号作为登录账号）。
  - **`password`**：非空。
  - **图形验证码**：**必填**；规则同 FR-AUTH-009。
- 提交成功后：用户处于**已登录**状态，并执行 FR-AUTH-006。

### FR-AUTH-003 跨页入口

- **登录页**必须提供清晰可见的入口，跳转至**注册页**（文案可「注册账号」等，由设计定稿）。
- **注册页**必须提供入口返回**登录页**（或「已有账号去登录」）。

### FR-AUTH-004 登出

- 系统应提供**登出**能力（可在导航、菜单或账户区域；具体位置由设计阶段定）。登出成功后：
  - 当前会话失效；
  - 用户访问需登录的资源时应被引导至登录流程（与 FR-AUTH-007 一致）。

> 若本迭代工程排期仅能保证「接口级登出 + 最小入口」，产品最低验收仍为：存在可触达的登出路径且会话被清除。

### FR-AUTH-005 错误与限制（用户可见行为）

| 场景 | 产品要求 |
| --- | --- |
| 邮箱格式错误 | 阻止提交或在失焦/提交时提示 |
| 凭证错误（密码错误或邮箱未注册） | 展示通用错误信息；**不得**在提示中泄露「邮箱是否已注册」的敏感枚举差异（推荐统一为「邮箱或密码错误」类表述，具体文案设计阶段定） |
| 邮箱已注册（注册提交） | 明确提示已注册，并引导去登录 |
| 手机号格式错误（已填写时） | 提示需 11 位数字，或允许留空 |
| 手机号已被他人在册占用（注册提交） | 明确提示占用，引导修改或留空 |
| 图形验证码错误 | 提示验证码错误，并支持刷新后重试，不消耗「登录失败次数」与邮箱密码错误混为一谈时可由实现约定 |
| 账号被禁用/冻结 | 展示不可用原因（若业务提供该状态） |
| 请求过于频繁 | 展示限流/稍后重试类提示，不暴露内部阈值 |

### FR-AUTH-006 会话与跳转期望

- **会话**：登录或注册成功后，须建立**服务端会话**（实现方式由后续服务端定义，如会话 Cookie）。产品期望：在会话有效期内，用户访问**首页 `/`**、**对话 `/chat`**、**控制台 `/console`** 时，不因「未登录」被无故拦截（除非另有产品规则）。
- **默认落地页**：登录/注册成功后的默认跳转目标为 **首页 `/`**。
- **可选**：若 URL 带合法 `redirect`（或等价参数）且目标为本站允许的路径白名单内，则优先跳转至该路径；若未带或非法，则回退默认首页。白名单与参数名可在实现阶段细化，本需求要求**存在可预测的「登录后去向」规则**并可被测试验证。

### FR-AUTH-007 受保护路由（产品级）

- **对话页 `/chat`** 与 **控制台 `/console`**：在未登录状态下，产品期望用户被**引导至登录**（或先展示需登录的说明再跳转），不得在无会话时展示仿佛已授权的业务数据。
- **首页 `/`**：允许公开访问或可展示未登录态；若选择「首页也需登录」，须单独修订 PRD；**默认假设首页可公开访问**，与 0.0.1「入口页」定位一致。

### FR-AUTH-008 安全与合规（产品级底线）

- 密码不得以明文形式出现在 URL、日志或可被第三方轻易读取的客户端存储中（工程实现约束）。
- 图形验证码须具备**刷新**与**服务端校验**，避免仅前端比对；具体算法与防 OCR 强度由实现阶段定义。

### FR-AUTH-009 图形验证码（本版本 In Scope）

- **范围**：登录页与注册页**均需**展示图形验证码（图片字符类或产品与设计共同确认的等价形态）。
- **不做**：短信 OTP、向邮箱发送一次性数字验证码完成登录/注册、邮箱链接激活（见 PRD 7.2）。
- **体验**：须支持用户看不清时**刷新**验证码；校验失败时给出明确提示，不泄露正确答案。

## 2. 字段与规则摘要

| 字段 | 规则（产品） |
| --- | --- |
| `email` | **必填**（注册）；**登录账号**；常见邮箱格式；系统内**唯一** |
| `telNo` | **可选**；若填写须为 **11 位数字**；若填写则系统内**唯一**（不被多账户共用） |
| `nickName` | 非空；建议长度 1–32 字符；不允许仅空白；非法字符集由实现约定 |
| `password` | **最少 8 位**；须**同时包含英文字母与数字**；不得与 `email` 完全相同（大小写规则由实现可选） |
| `passwordConfirm` | 须与 `password` 一致 |
| `captcha` / 等价字段 | 与当次图形验证码绑定；由服务端校验 |

## 3. 密码策略（确定版）

- 长度：≥ 8。
- 复杂度：至少包含 **字母（A–Z 或 a–z）** 与 **数字（0–9）** 各至少一个。
- 不允许：与 **邮箱全文** 完全相同。

## 4. 短信 / 邮箱一次性验证码

- **本版本不做**：短信 OTP、向邮箱发送 6 位验证码、邮箱验证链接完成注册。
- **与图形验证码区分**：本版本**做**图形验证码（见 FR-AUTH-009）。

## 5. 非目标（防范围蔓延）

- 不包含第三方社交登录、企业 SSO。
- 不包含找回密码、换绑手机、换绑邮箱、账号注销（可列入路线图）。
- 不包含基于角色的控制台权限细分。
- 不包含对话与控制台内部业务功能，仅认证与路由可达性。

## 6. 验收标准（Acceptance Criteria）

- [ ] **AC-01** 存在可访问的**注册页**与**登录页**；登录页不再是「仅占位不可提交」的产品定义。
- [ ] **AC-02** 注册流程**必填邮箱**；手机号**可不填**；若填手机号须 11 位数字；含昵称、密码与确认密码；规则不符合时无法完成成功注册或须明确报错。
- [ ] **AC-03** 登录流程使用 **邮箱 + 密码**；错误凭证下不能进入已登录态。
- [ ] **AC-04** 重复 **邮箱** 注册时提示已注册，并引导至登录。
- [ ] **AC-05** 登录页与注册页均包含**图形验证码**，且错误/刷新行为可测。
- [ ] **AC-06** 登录/注册成功后用户处于已登录会话，默认进入 **`/`**；在约定 `redirect` 规则下可进入白名单内页面。
- [ ] **AC-07** 未登录访问 `/chat`、`/console` 时，产品定义的拦截/引导行为成立（见 FR-AUTH-007）。
- [ ] **AC-08** 登出后会话清除，再访问受保护路由需重新登录。
- [ ] **AC-09** 文档与实现确认：不包含短信/邮箱 **OTP** 作为登录或注册手段；图形验证码作为本版本范围。

## 7. 追溯与变更

- 本需求承接 0.0.1 页面与路由命名；取代 0.0.1 中对登录页「不得实现认证」的约束（新约束以 0.0.2 为准）。
- **变更说明（相对早前 0.0.2 草案）**：登录账号由手机号改为**邮箱**；注册**必填邮箱**、**手机号可选**；**图形验证码**纳入本版本范围。
