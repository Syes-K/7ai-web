# PRD - 登录与注册能力（认证入口）- version 0.0.2

## 1. 文档信息

| 项 | 内容 |
| --- | --- |
| 版本 | 0.0.2 |
| 类型 | 产品需求（阶段 1） |
| 前置版本 | 0.0.1（四页面静态占位；登录页为占位、无真实认证） |
| 视觉 | 本阶段不产出视觉稿；整体体验后续由设计定为「赛博 & 科技黑」风格，仅作背景说明，不约束本 PRD 的验收项 |

## 2. 背景

0.0.1 已确立信息架构与路由雏形（首页、登录、对话、控制台），并明确**不实现**鉴权与会话。为进入可用的账号体系，本迭代需在**产品层面**定义「登录」与「注册」的完整需求边界、字段与规则、成功后的会话与跳转期望，供设计（阶段 2）与后续服务端/前端实现对齐。

## 3. 目标

- 为新用户提供**注册**路径，为老用户提供**登录**路径，形成最小闭环的账号密码认证能力：**登录账号标识为邮箱**；注册须**必填邮箱**，**手机号可选**；与昵称、密码等字段共同构成账户信息。
- **登录页与注册页均须提供图形验证码**（人机校验，具体交互形态由设计定），以降低自动化撞库与批量注册风险。
- 明确错误、频控与安全相关**产品级**期望，避免实现阶段口径不一。
- 明确登录成功后的**会话存在形式**与**默认跳转/可达路由**，与 0.0.1 已存在的页面路由衔接。
- 通过「非目标」控制范围，避免本迭代承担 OAuth、短信/邮箱**一次性验证码登录或注册**等扩展能力（与「图形验证码」区分）。

## 4. 目标用户

- **新用户**：首次使用产品，需完成注册后登录。
- **老用户**：已拥有账号，仅需登录。
- **项目团队**：产品、设计、前后端以本文档与 `requirements.md` 为共同基线。

## 5. 核心场景

1. **新用户注册**：访问注册页 → 填写**邮箱（必填）**、**手机号（可选）**、昵称、密码 → 完成**图形验证码** → 提交 → 成功后进入已登录状态并跳转至约定落地页。
2. **老用户登录**：访问登录页 → 填写**邮箱**、密码 → 完成**图形验证码** → 提交 → 成功后进入已登录状态并跳转至约定落地页。
3. **错误与限制**：凭证错误、格式错误、账号不可用、重复注册、请求过于频繁等场景下，用户能感知原因并有机会修正或稍后重试（具体交互由设计阶段细化，本阶段定义**可验收**的行为与信息层级）。
4. **登出（若纳入本版本）**：用户主动结束会话；登出后受保护页面不可再以当前身份访问（见范围说明）。

## 6. 成功指标（产品）

- 需求文档可被测试用例直接映射：**注册、登录、主要错误与限制、登录后跳转与会话**均有明确验收标准。
- 字段与规则（含密码策略、图形验证码是否在范围）在文档中**唯一确定**，减少返工。
- 非目标清单清晰，本迭代不包含 OAuth、短信/邮箱**一次性验证码**用于登录或注册（与图形验证码不同）；除非在范围修订中单独立项。

## 7. 范围

### 7.1 本次要做（In Scope）

- **页面/路由（产品要求）**
  - 登录页：与现有 `/login` 衔接，从占位升级为**可提交**的登录流程（实现归属后续阶段）。
  - 注册页：新增注册入口与独立注册页路由（建议 `/register`，具体以设计与实现为准，本需求以「存在可访问的注册页」为验收）。
- **字段与业务规则（产品级）**
  - **邮箱 `email`（必填）**：用于注册与**登录账号**；格式与唯一性见 `requirements.md`。
  - **手机号 `telNo`（可选）**：若填写则须为 **11 位数字**；不填则跳过手机号校验。
  - **昵称 `nickName`、密码与确认密码**：见 `requirements.md`。
  - **密码策略**：长度与复杂度下限（见 `requirements.md`）。
  - **图形验证码**：登录与注册流程**均需**；刷新、错误提示与防刷策略见 `requirements.md`（不含短信/邮箱 OTP）。
- **会话与跳转**
  - 登录/注册成功后：建立**服务端认可的会话**（具体技术实现由后续服务端文档定义；产品侧要求「刷新或关闭页签后，在合理时间内仍保持登录态」的期望见 `requirements.md`）。
  - 默认落地：与 0.0.1 路由衔接，成功后可进入**首页 `/`**；用户应能继续访问 **对话 `/chat`**、**控制台 `/console`**（是否默认需登录由「受保护路由」规则定义，见 `requirements.md`）。
- **错误与限制（产品级）**
  - 无效凭证、格式错误、重复注册、账号不可用、频控等均有对应用户可见结果与可测试标准。

### 7.2 本次不做（Out of Scope）

- **视觉与交互稿**：赛博 & 科技黑等风格由设计阶段产出，本阶段不要求像素级规范。
- **短信 / 邮箱一次性验证码**：不包含短信 OTP、向邮箱发送 6 位验证码完成注册或登录、邮箱链接激活等（与「图形验证码」不同）；若未来引入，须新版本 PRD 单独立项。
- **其他第三方人机**：不包含依赖 Google reCAPTCHA v3 等需外站账号的深度集成，除非与「图形验证码」方案合并评估后单独立项。
- **第三方与扩展登录**：不包含微信/Apple/Google 等 OAuth、企业 SSO。
- **找回密码 / 修改手机号 / 账号注销**：不包含（可列入路线图，非 0.0.2）。
- **角色与权限体系**：不包含管理员与普通用户的细粒度 RBAC（与 0.0.1 后端草案「控制台暂不区分角色」方向一致，后续单独立项）。
- **对话与控制台业务功能**：不包含；仅要求「已登录用户可访问对应路由」层面的产品期望。

## 8. 与 0.0.1 / 上游草案的衔接说明

- **路由**：延续 0.0.1 已存在的首页、登录、对话、控制台路径；本版本**新增**注册相关路由需求。
- **接口方向（非本阶段实现约束）**：0.0.1 `api-spec` 草案中已有 `POST /api/auth/login` 等方向；本迭代以 **`email` 为登录账号** 为准，后续 API 文档须将请求体中的账号字段与 `email` 对齐（若曾使用 `account` 命名，需注明语义为邮箱）。`telNo`、`nickName` 等与 `requirements.md` 保持一致。

## 9. 依赖与假设

- 依赖后续阶段提供：服务端会话方案、密码哈希与存储、频控实现方式；本产品文档不规定技术选型。
- 假设生产环境将使用 HTTPS；明文传输风险由工程规范覆盖，不单列产品验收项。
- 假设同一 **邮箱** 全局唯一；重复注册按「该邮箱已注册」类错误处理。若用户填写手机号，则同一手机号在未解绑前提下不得被多账户占用（见 `requirements.md`）。

## 10. 风险与待确认（产品侧）

| 风险/项 | 说明 | 处理 |
| --- | --- | --- |
| 会话时长与「记住我」 | 是否区分长期会话与浏览器关闭即失效 | 默认不单独做「记住我」产品项；统一会话策略见 `requirements.md` |
| 注册页入口位置 | 登录页是否必须展示「去注册」链 | In Scope：登录页提供前往注册入口 |

## 11. 待设计介入项（交接给阶段 2）

- 登录页、注册页的布局、组件层级与赛博 & 科技黑视觉语言。
- 错误态、加载态、禁用态与表单校验提示的呈现方式（需满足 `requirements.md` 中信息可见性要求）。
- **图形验证码**的布局、刷新入口、失败态与无障碍可读性（在不泄露安全细节前提下可测）。
- 注册页与登录页之间的导航关系（入口位置、返回行为）。
