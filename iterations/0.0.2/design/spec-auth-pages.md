## 3. 登录页 `/login`

### 3.1 页面目标

已注册用户通过 **邮箱 + 密码 + 图形验证码** 进入已登录态；提供前往注册的清晰入口。

### 3.2 信息结构（自上而下）

1. **页面主标题**（H1）：「登录」或产品选定简短标题（与 0.0.1 页面身份一致即可）。
2. **可选副文案**：一句中性说明（如「使用邮箱登录你的账号」），`type.small`，`text.secondary`。
3. **全局/服务端错误区**（条件显示）：卡片内主按钮上方或标题下，见 §5。
4. **表单字段**（顺序固定）：
   - **邮箱** `email`：类型 `email`，自动完成 `username`（实现可选）。
   - **密码** `password`：`type="password"`，可配「显示/隐藏」图标按钮（可选增强）。
   - **图形验证码**：见 §6。
5. **主操作**：按钮文案建议「登录」；加载态见 §3.4。
6. **次要操作区**：
   - **前往注册**：文案建议「**没有账号？去注册**」或「注册账号」；链接样式 `accent.primary`，与 FR-AUTH-003 一致。

### 3.3 组件清单

| 组件 | 说明 |
| --- | --- |
| `AuthPageShell` | 背景 + 顶栏 + 居中卡片容器（见 `design-spec.md` §5） |
| `AuthCard` | 卡片容器与标题区 |
| `TextField` | 邮箱、密码 |
| `CaptchaField` | 图片 + 刷新 + 文本输入（§6） |
| `PrimaryButton` | 提交 |
| `TextLink` | 跳转注册 |

### 3.4 状态矩阵（登录）

| 状态 | UI 行为 | 备注 |
| --- | --- | --- |
| 默认 | 提交按钮可点（若实现「空字段禁用」则邮箱、密码、验证码均非空才可点，二选一由前端与产品共识；**设计默认**：允许点提交再由客户端提示，避免过度静默） | — |
| 客户端校验 | 邮箱格式错误：失焦或提交时在字段下显示短文案（§5.1）；阻止提交或可提交由实现定，**错误须可见** | 对齐 FR-AUTH-005 |
| 提交中（loading） | 主按钮内 spinner + 文案「登录中…」；按钮 `disabled`；可选禁用整个表单防重复 | US-02 |
| 凭证错误（服务端） | **不区分**邮箱未注册与密码错误：展示 **§5.2 登录凭证统一文案**；可清空验证码并刷新 | **FR-AUTH-005** |
| 验证码错误 | 明确提示「验证码错误」或「验证码不正确」；**自动刷新**验证码图或引导用户点刷新；**不与**「邮箱或密码错误」混为一条模糊文案 | FR-AUTH-005、AC-US-02-3 |
| 账号禁用 | 单独文案（若业务返回）；可用 `state.danger` 条 | FR-AUTH-005 |
| 限流 | 卡片级或全局提示，`state.warning`，见 §5.3 | US-03 |
| 禁用（维护等） | 整页或按钮级禁用 + 说明（极少用） | — |

### 3.5 与 US-02 AC 对齐要点

- **AC-US-02-2**：仅使用 §5.2 统一凭证错误表述。
- **AC-US-02-3**：验证码错误与凭证错误在文案与刷新行为上可区分（至少一种：不同文案或验证码自动刷新）。
- **AC-US-02-5**：「去注册」在首屏可见，不低于主按钮的视觉权重过多稀释（置于主按钮下即可）。

## 4. 注册页 `/register`

### 4.1 页面目标

新用户提交 **必填邮箱**、**可选手机号**、昵称、密码、确认密码及图形验证码；成功则已登录并跳转（FR-AUTH-006）。

### 4.2 信息结构（自上而下）

1. **页面主标题**（H1）：「注册」或「创建账号」。
2. **可选副文案**：如「使用邮箱注册；手机号选填」。
3. **全局/服务端错误区**（条件显示）。
4. **表单字段**（顺序建议）：
   - **邮箱** `email`（必填）
   - **手机号** `telNo`（可选）：placeholder 建议「选填，11 位数字」；留空不校验格式
   - **昵称** `nickName`（必填）
   - **密码** `password`（必填）
   - **确认密码** `passwordConfirm`（必填）
   - **图形验证码**（§6）
5. **主操作**：「注册」或「创建账号」；加载态「注册中…」。
6. **次要操作**：**已有账号？去登录** → `/login`（FR-AUTH-003）。

### 4.3 组件清单

在 §3.3 基础上增加昵称、手机号、确认密码字段；手机号表单项旁可用 `type.small` 说明「选填」。

### 4.4 状态矩阵（注册）

| 状态 | UI 行为 |
| --- | --- |
| 默认 | 同登录 |
| 客户端校验 | 邮箱格式、手机号 11 位（若填写）、密码策略（长度≥8、字母+数字、不得与邮箱全文相同）、两次密码一致；逐项字段下错误 | FR 字段规则、US-03 |
| 提交中 | 同登录，文案「注册中…」 |
| 邮箱已注册 | **明确**提示该邮箱已注册 + **主链**「去登录」按钮或醒目链接 | FR-AUTH-005（与登录凭证枚举差异区分） |
| 手机号被占用 | 明确提示占用，建议修改或留空 | FR-AUTH-005 |
| 验证码错误 | 同登录 §3.4；刷新后重试 |
| 限流 / 禁用 | 同登录 |

### 4.5 与 US-01 AC 对齐要点

- **AC-US-01-5**：与登录页的「凭证统一错误」不同，注册场景**允许且必须**明确「邮箱已注册」（产品已区分两种场景的信息策略）。

## 5. 文案与信息层级（含 FR-AUTH-005）

### 5.1 客户端校验（建议文案）

| 场景 | 建议对外文案 | 展示位置 |
| --- | --- | --- |
| 邮箱格式无效 | 「请输入有效的邮箱地址」 | 邮箱字段下 |
| 手机号已填但非法 | 「手机号需为 11 位数字」或「请输入 11 位数字手机号」 | 手机字段下 |
| 密码不符合策略 | 「密码至少 8 位，且需同时包含字母与数字」；若与邮箱相同则加一句「密码不能与邮箱相同」 | 密码字段下 |
| 确认密码不一致 | 「两次输入的密码不一致」 | 确认密码字段下 |
| 必填缺失 | 「请填写xxx」 | 对应字段下 |

### 5.2 登录 — 凭证类服务端错误（FR-AUTH-005）

- **必须**：对「密码错误」与「邮箱未注册」**不区分**对外展示。
- **推荐主文案（唯一对外句）**：**「邮箱或密码错误，请检查后重试。」**
- **辅助行为**：刷新或清空图形验证码（若服务端在凭证错误时也作废验证码）；**不要**出现「该邮箱未注册」「邮箱不存在」等表述。
- **层级**：`state.danger` 行或 Alert；可加 `type.caption` 一句「如忘记密码，请联系支持（若产品未上找回密码则不要写「找回密码」链接）」——**0.0.2 无找回密码**，建议**不写**占位链接，仅保留主句即可。

### 5.3 登录 / 注册 — 验证码错误

- **主文案**：「验证码错误，请重新输入」或「验证码不正确」。
- **推荐行为**：展示错误同时 **自动加载新验证码**；若仅手动刷新，则错误文案中加「请点击刷新」。
- **与凭证错误区分**：勿将验证码错误写成「邮箱或密码错误」。

### 5.4 注册 — 邮箱已注册

- **主文案**：「该邮箱已注册。」
- **操作**：次要按钮样式或链接「去登录」→ `/login`（可带 `redirect` 参数规则由实现定）。

### 5.5 限流

- **主文案**：「请求过于频繁，请稍后再试。」
- **可选**：`type.caption` 显示大致等待时间（若后端提供）。

### 5.6 账号不可用（禁用/冻结）

- 使用业务返回的可展示原因短句；无则通用：「账号暂不可用，请联系支持。」

## 6. 图形验证码区块（两页共用）

### 6.1 结构

- **行布局**：左侧验证码图片容器 + 刷新按钮；右侧或下方为 **输入框**（标签建议「验证码」）。
- **标签**：可见文字「验证码」；刷新按钮 `aria-label="刷新验证码"`。
- **辅助说明**（`type.caption`，`text.secondary`）：「看不清？点击刷新换一张。」——满足可理解性，**音频等替代方案**见 `design-spec.md` §8 原则（后续工程）。

### 6.2 交互

- **刷新**：点击后进入短 loading（图片区），成功后替换图片；失败则 toast/行内「刷新失败，请重试」。
- **错误后**：输入框可保留用户输入或清空（实现择一，建议清空验证码输入以减少误提交）。

### 6.3 与登录失败计数

- 产品允许验证码错误与邮箱密码错误在「是否计入撞库计数」上由实现约定；**设计侧**：验证码错误提示中**不出现**「密码错误次数」类文案，避免与 FR-AUTH-005 混淆。

## 7. 成功后的跳转（设计侧期望）

- 默认进入 **首页 `/`**；若 URL 带合法 `redirect`，优先跳转白名单内页面（FR-AUTH-006）。
- **视觉**：成功态可仅用「按钮 loading 结束 → 路由切换」，无需额外全屏成功页。

---

## 8. 登出（US-05）与认证双页关系

- **FR-AUTH-004 / US-05**：登出入口可在导航、菜单或账户区；**不要求**放在登录/注册页。
- 本版若顶栏为全局组件，可在**已登录**态显示用户菜单内含「登出」；登录/注册页面对**未登录**用户展示即可。

## 9. 页面间导航关系小结

| 从 | 到 | 入口位置 | 文案建议 |
| --- | --- | --- | --- |
| 登录 | 注册 | 主按钮下方次要区 | 没有账号？**去注册** |
| 注册 | 登录 | 主按钮下方次要区 | 已有账号？**去登录** |

两链均使用 `TextLink` 样式，保证在 `design-spec.md` 的霓虹对比下可辨识、可聚焦。
