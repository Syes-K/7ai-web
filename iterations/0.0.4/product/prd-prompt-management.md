# PRD：管理后台提示词配置（`/admin/prompts`）

**版本**：`0.0.4`  
**路由**：`/admin/prompts`（侧栏「提示词管理」）  
**关联代码**：`src/common/constants/defautPromptConfig.ts` → `DEFAULT_PROMPT_CONFIG`  
**数据文件（约定）**：项目根 `data/promptConfig.json`

---

## 1. 需求摘要

### 1.1 背景

管理后台已具备 `/admin` 及「提示词管理」入口，当前页面为占位。业务上需要将「可配置的提示词模板」从代码常量中**可运营化**：在保留代码内默认值作为兜底的前提下，允许通过 JSON 文件覆盖部分或全部配置项，并在管理后台以表单形式查看与编辑。

### 1.2 目标用户

- 已登录、可访问管理后台的内部运营 / 技术人员（与现有全站会话保护一致）。

### 1.3 核心场景

1. 打开「提示词管理」页，看到当前生效的各配置项（标签来自 `name`，说明来自 `desc`，正文为 `value`）。
2. 修改某项 `value`（及在允许范围内调整展示用元数据，若产品允许）并保存，使后续运行时读取到新配置。
3. 当 `data/promptConfig.json` 缺失、损坏或缺少某 key 时，系统对该 key 仍可使用 `DEFAULT_PROMPT_CONFIG` 中的默认值，管理页应能展示**合并后的有效配置**，避免空白或报错中断使用。

### 1.4 成功指标（可验收）

- 管理页能列出所有**当前版本产品约定的配置 key**（见范围），且展示字段符合：`name` 为表单项标签、`desc` 为标签旁 Tooltip、`value` 为多行正文编辑区。
- 读取逻辑满足：**按 key 自 `data/promptConfig.json` 读取；缺 key 或文件不存在时用 `DEFAULT_PROMPT_CONFIG` 对应项补齐**（合并结果与业务读取侧一致）。
- 保存行为在实现阶段按本文「假设」落地后，刷新页面或重新进入仍能看到持久化结果（若采用写回 JSON 方案）。

## 2. 功能范围

### 2.1 本期做（In Scope）

| 编号 | 能力 | 说明 |
|------|------|------|
| F1 | 配置合并读取 | 以 `DEFAULT_PROMPT_CONFIG` 的 key 集合为**权威 key 列表**；对每个 key，若 `data/promptConfig.json` 中存在且可解析为合法片段，则覆盖默认的 `name`/`desc`/`value` 中实际提供的字段；文件缺失或某 key 缺失则该 key 完全采用默认值。 |
| F2 | 管理页表单 | 表单项标签 = `name`；标签右侧或旁提供 Tooltip，内容为 `desc`（支持较长说明）；`value` 为提示词正文编辑器（支持多行、保留占位符如 `{{content}}` 的文本形式）。 |
| F3 | 与现有鉴权一致 | `/admin` 及子路由继续受现有会话 Cookie 与中间件保护；未登录用户跳转登录（与 `middleware` 行为一致）。 |
| F4 | 基础数据校验 | 保存或提交时校验：每项须满足结构约定（见下）；`value` 非空等业务规则可按实现约定；非法 JSON 文件需有明确错误提示策略（见假设）。 |

**单项结构约定（与常量一致）**

- 每个 key 对应对象字段：`name`（string）、`desc`（string）、`value`（string）。

### 2.2 本期不做（Out of Scope）

| 编号 | 不做 | 说明 |
|------|------|------|
| O1 | 任意新增/删除配置 key | 不以管理 UI 动态增删 key；key 集合以代码中 `DEFAULT_PROMPT_CONFIG` 为准（后续若扩展 key，通过发版改常量）。 |
| O2 | 多语言 UI | 管理页界面语言跟随现有 admin，不单独做 i18n。 |
| O3 | 版本历史与回滚 | 不提供内置变更历史、对比与一键回滚（可用 Git 管理 `data/promptConfig.json`）。 |
| O4 | 富文本 / 变量自动插入 | `value` 按纯文本编辑；不强制提供占位符可视化插入器（可作为设计增强，非必须）。 |

### 2.3 依赖与假设

- 依赖 Next.js 应用可访问项目根下 `data/` 目录（服务端读写或构建时打包策略由设计与后端阶段确定）。
- 假设管理后台使用者具备理解「占位符」含义的能力；占位符语法不与本文冲突，保持字符串原样存储即可。

## 3. 用户故事与验收标准

### US-1 查看合并后的提示词配置

**作为** 已登录的管理后台用户  
**我想要** 在「提示词管理」页看到所有预定义 key 的当前生效配置（含来自 JSON 的覆盖与默认值补齐）  
**以便** 确认线上或本地将使用的提示词内容

**验收标准：**

- [ ] AC1：页面展示的 key 集合与当前 `DEFAULT_PROMPT_CONFIG` 的 key 一致（camelCase）。
- [ ] AC2：当不存在 `data/promptConfig.json` 时，各表单项展示的值与 `DEFAULT_PROMPT_CONFIG` 完全一致。
- [ ] AC3：当 JSON 仅覆盖部分 key 时，未出现的 key 展示默认值；已出现且字段合法的 key 展示合并后的结果。
- [ ] AC4：`name` 显示为表单项标签，`desc` 可通过 Tooltip 查看完整说明，`value` 在可编辑区域展示。

---

### US-2 通过表单编辑提示词正文

**作为** 管理后台用户  
**我想要** 在表单中修改某项的 `value`（必要时也可调整 `name`/`desc` 若产品允许写回）  
**以便** 快速迭代提示词而无需改代码

**验收标准：**

- [ ] AC1：`value` 支持多行输入；保存前后占位符（如 `{{content}}`）不被意外转义或剥离。
- [ ] AC2：提交前有基础校验（空 `value`、类型错误等）并给出用户可理解的错误提示（具体规则可与后端约定）。
- [ ] AC3：保存成功后用户能收到成功反馈；再次进入页面应看到更新后的内容（与持久化方案一致）。

---

### US-3 异常与边界下的可读性

**作为** 管理后台用户  
**我想要** 在 JSON 文件损坏或部分字段非法时仍能进入页面或使用合理降级  
**以便** 排查问题而不是整页白屏

**验收标准：**

- [ ] AC1：`data/promptConfig.json` 无法解析时，有明确错误提示，且对各 key 至少可回退展示 `DEFAULT_PROMPT_CONFIG`（或等价安全策略，见待确认）。
- [ ] AC2：JSON 中某 key 的对象缺少 `name`/`desc`/`value` 时，缺失字段用默认值补齐，不导致整表失败（与 F1 合并规则一致）。

## 4. 待设计项清单（交接设计阶段）

以下项需设计产出：布局、组件选型（与现有 Ant Design / ProComponents 一致）、交互状态与文案。

| ID | 项 | 说明 |
|----|----|------|
| D1 | 页面信息架构 | `PageContainer` 下表单整体布局：单列 vs 分组；长 `value` 的输入高度与折叠策略。 |
| D2 | Tooltip 交互 | `desc` 较长时的 Tooltip 触发方式（hover / 问号图标）、最大宽度与滚动。 |
| D3 | 表单项与 key 的对应关系 | 是否在 UI 上展示只读的「配置 key」（camelCase）便于排查与文档对照（建议次要展示）。 |
| D4 | 保存操作 | 主按钮「保存」位置、loading、禁用态；是否「全部保存」一次提交或分 key 保存（建议整表一次保存以降低并发写文件风险）。 |
| D5 | 错误与空状态 | JSON 解析失败、权限不足、磁盘写入失败时的反馈样式与文案层级。 |
| D6 | 占位符说明 | 可选：在页面底部或每项下增加简短说明「支持的占位符」列表（若仅 `{{content}}` 可写死示例）。 |

## 5. 待确认与假设（产品默认倾向）

以下在需求中未完全指定，**默认倾向**已写明；实施前建议产品/技术确认。

| 主题 | 默认假设 | 待确认问题 |
|------|-----------|------------|
| 持久化 | **保存时写回** `data/promptConfig.json`：以合并后的完整对象或「仅含覆盖项」的子集写入，需与运行时读取规则一致（若只写覆盖项，读取侧须继续与默认合并）。 | 是否允许写回；若部署环境为只读文件系统，是否改为仅内存/外部配置服务。 |
| 可编辑字段 | **优先仅强需求编辑 `value`**；`name`/`desc` 是否允许在 UI 修改并写回 JSON（会改变运营改文案而不改代码的路径）。 | 是否锁定 `name`/`desc` 只读，仅编辑 `value`。 |
| 运行时生效 | 文件变更后，**不保证**无需重启即可被所有进程感知；以实际部署方式为准。 | 是否需要文档说明「部署后重启」或实现文件监听热读。 |
| 权限与审计 | 与当前 **admin 登录** 同级，不单独区分角色；**无**操作审计日志。 | 是否需要管理员角色细分、操作人日志。 |
| JSON 校验 | 保存前在服务端校验结构与类型；读取时坏文件：**倾向**提示错误 + 全量回退默认（或仅拒绝写、读仍合并），需统一一种策略。 | 坏文件时是「禁止保存 + 只读展示默认」还是「告警但可编辑覆盖修复」。 |
| 多环境 | `data/promptConfig.json` 是否进版本库：建议 **gitignore 或提供 `*.example`**，由环境自行放置。 | 与运维约定一致即可。 |
