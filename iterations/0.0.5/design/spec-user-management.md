## 页面与信息架构

### 整体结构

- 页面置于既有 **admin layout** 内：侧栏选中「用户管理」，内容区为 **`/admin/users`** 子路由。
- 建议使用 **`PageContainer`**（与 `0.0.3` 及 `prompts` 等页一致）：`title` = **用户管理**；可选 `subTitle` 一句说明：「系统账号运维：查看状态、调整可用性、重置密码。」（实现可省略副标题以控复杂度）。

### 内容区自上而下

| 区域 | 说明 |
| --- | --- |
| **页头** | 标题 +（可选）简短说明；**不与** `/console` 共用「控制台」类文案 |
| **工具栏** | 左侧：**关键字搜索**（见 §8）；右侧：**刷新**按钮（重新请求当前页列表） |
| **主列表** | **表格**（ProTable 或 Table + 自建请求，与工程惯例一致）；行内操作为主（见 §5） |
| **底部分页** | 与表格一体的分页器：**页码** + **每页条数**（若后端支持 `pageSize`）；显示总条数或「第 x–y 条」等以后端为准 |

### 导航与心智

- 本页属于 **管理后台 / 系统管理**，文案避免与普通用户「我的账号」自助流程混同。
- 深链 `/admin/users` 应直接展示本页（与 `0.0.3` 路由表一致）。

## 表格列建议

列宽与排序优先级：**桌面首屏**保证「标识 + 状态 + 锁定 + 操作」可见；其余列可次要或按需横向滚动。

| 列（建议顺序） | 内容说明 | 与故事 / AC 对应 |
| --- | --- | --- |
| **用户标识** | 主展示：**邮箱或用户名**（以后端字段为准，二选一或组合一行内主次两行）；可选副信息：用户 ID 小号字/次要列 | 3.1 AC3（可区分用户的标识） |
| **状态** | 用户账号**业务状态**（启用 / 停用 / 等枚举，标签 `Tag` 着色：**成功色=可用**、**默认/警告=受限** 等，以后端枚举映射为准） | 3.1 AC3、3.2 |
| **锁定** | **锁定相关**（只读）：如「正常 / 已锁定」+ 可选剩余时间（若后端提供）；**不提供**行内解锁按钮 | 3.1 AC3 |
| **最近更新**（可选） | `updatedAt` 或等价字段，便于运维排查 | 支撑列表运维场景（PRD 1.3） |
| **操作** | **行内**文字按钮或 `Button type="link"`：**状态变更**（随当前状态切换「启用」「停用」文案）、**重置密码**；按钮按 §5 与 §6 显示/禁用 | 3.2、3.3 |

**操作列规则（摘要）**

- **状态变更**：若当前状态不允许某种转换，对应入口 **禁用** 或隐藏（与 PRD 3.2 AC1 一致）。
- **重置密码**：始终可见（若业务允许对所有可管理用户重置）；若后端禁止对某类用户操作，则 **禁用 + Tooltip 说明**。

## 交互与流程

### 通用约定

- 所有破坏性/高风险操作（**停用**、**重置密码**）在发起请求前使用 **`Modal.confirm`**（或 antd `App` 体系下等价的 `modal.confirm`）做**二次确认**。
- 确认框正文**必须包含目标用户标识**（与 PRD 各节 AC 一致），与表格主标识展示一致，避免误操作。
- **危险视觉**：重置密码、停用账号等使用 **`okType: 'danger'`**（或主按钮 `danger`），与「启用」等正向操作区分。

### 5.1 状态变更（启用 / 停用）

| 项 | 建议文案 |
| --- | --- |
| **标题** | `{启用|停用}用户账号？`（动态随操作变化） |
| **正文** | `将对「{用户标识}」执行「{启用|停用}」。停用后该用户将无法登录系统（具体以系统策略为准）。` |
| **主按钮** | `确认{启用|停用}`（停用 → **danger**） |
| **次按钮** | `取消` |
| **成功后** | `message.success` 简短成功文案；**刷新当前行或整表**（与 PRD 3.2 AC3 一致） |
| **失败后** | `message.error` 展示后端返回信息 |

### 5.2 重置密码

| 项 | 建议文案 |
| --- | --- |
| **标题** | `重置该用户的密码？` |
| **正文** | `将为「{用户标识}」生成新的登录密码。请仅在安全渠道告知用户；请勿在公共场合展示。` |
| **主按钮** | `确认重置`（**danger**） |
| **次按钮** | `取消` |
| **成功后** | 见 **§7 方案 A**（独立 Modal，非仅 toast） |
| **失败后** | `message.error`（频率限制、策略不允许等） |

### 批量操作

- **本迭代不做**行选择列与批量操作（与 PRD 2.2、D3 一致）。

## 状态完整（全链路）

| 状态 | 表现 |
| --- | --- |
| **默认 / 首屏** | 进入页面即加载第一页列表；工具栏与表格结构稳定，避免布局跳动 |
| **加载中** | 表格区域使用 **`Spin` 包裹**或 **表格 `loading`**；与 `admin/prompts` 页类似，避免白屏；**不**用假数据填充 |
| **空列表** | 表格内 **`Empty`**：标题如「暂无用户」；说明「当前条件下没有可展示的用户」；可选「刷新」 |
| **列表请求失败** | 顶部或表格上方 **`Alert` type="error"`** + 文案「用户列表加载失败」；提供 **「重试」** 按钮重新请求 |
| **无权限（403）** | 与现有 admin API 行为对齐：可 **`message.error`** 后 **replace 到 `/console?notice=admin_forbidden`**（与 `prompts` 页模式一致），**不**展示可操作数据（PRD 3.1 AC5） |
| **未登录（401）** | **redirect** 至 `/login?redirect=/admin/users`（与壳规范一致） |
| **操作进行中** | 确认框关闭后，全局或行内 **loading**（防止重复点击）；按钮 **disabled** |
| **操作成功** | `message.success`（除密码重置成功见 §7）；列表与操作列与最新状态一致 |
| **操作失败** | `message.error` + 后端 message；列表保持旧状态 |

## 密码重置成功态（本迭代锁定）

### 决策

| 项 | 内容 |
| --- | --- |
| **PRD 方案** | A：服务端生成随机**临时密码**，界面**一次性展示** |
| **本迭代主路径** | **锁定为方案 A** |
| **依据** | 当前工程扫描**未发现**可复用的邮件发送基础设施（无 nodemailer / SMTP 等）；PRD 5.1 建议无邮件时采用 A。若后续迭代接入邮件，可再增加方案 B 的 UI 分支，本文档不阻塞 backend 预留接口。 |

### 界面说明（方案 A）

1. 用户在前序 **`Modal.confirm`** 中确认「确认重置」后，请求成功时**关闭确认框**。
2. 立即打开 **`Modal`**（非 confirm，可关闭但见下条安全约束）：
   - **标题**：`密码已重置`
   - **正文区**：
     - 说明：`「{用户标识}」的临时密码如下。此密码仅显示一次，关闭窗口后请通过安全渠道告知用户。`
     - **临时密码**：等宽字体、可选 `Input.Password` 只读展示或纯文本 + **「复制」** 主按钮。
     - **安全提示**：`Alert` type="warning"` 或简短文案：「请勿通过即时通讯公群发送；建议用户首次登录后立即修改密码。」
   - **主按钮**：`复制密码并关闭`（先复制到剪贴板，再关闭 Modal）；**次按钮**：`关闭`（无复制则仅关闭）。
3. **关闭后**：页面上**不保留**临时密码明文；用户若需再次获取须重新执行重置流程（与 PRD 3.3 AC4、密码泄露风险缓解一致）。
4. **复制能力**：使用 Clipboard API；失败时 `message.warning` 提示手动复制。

### 与方案 B 的关系

- **本迭代 UI 不实现**「已发送重置邮件」为主的成功态；若 backend 未来返回「邮件已发送」，可在后续版本增加分支，**不在 0.0.5 范围**。

## 筛选 / 排序（MVP）

| 能力 | 本迭代 |
| --- | --- |
| **分页** | **做**：页码、每页条数（以后端为准） |
| **关键字搜索** | **做（建议）**：工具栏单一输入框，占位符如「搜索邮箱或用户名」；**回车或点击搜索**触发列表请求；**清空**恢复无关键字的第一页 |
| **多条件筛选**（角色、状态多选、时间范围等） | **不做**（PRD 2.2） |
| **列排序**（点击表头排序） | **不做**（除非 backend 已支持且成本极低；默认不做） |
| **刷新** | **做**：工具栏右侧刷新当前查询条件与页码 |

**扩展位**：工具栏右侧预留 **「筛选」** 占位**不强制**；若不做筛选折叠 UI，则仅搜索 + 刷新即可。

## 响应式

- **桌面优先**（与 PRD、0.0.3 壳一致）：主要运维场景在 **≥ 992px** 下完整展示。
- **窄屏 / 小宽度**：表格容器 **`overflow-x: auto`**，**横向滚动**查看全部列；**不做**本迭代内的卡片化逐行视图（PRD D7 可选项明确**不做**）。

## 可访问性（简要）

| 项 | 建议 |
| --- | --- |
| **焦点顺序** | 工具栏从左到右（搜索 → 刷新）→ 表格按行 → 行内操作从左到右；分页在表格后聚焦 |
| **确认对话框** | 使用 antd Modal 默认焦点陷阱；打开时焦点在**主按钮**或**安全的主操作**；关闭后焦点回到触发按钮（若框架支持 `focusTriggerAfterClose`） |
| **表格操作** | 行内按钮带清晰文案；若仅用图标，必须 `aria-label`（本迭代推荐**文字按钮**降低歧义） |
| **状态与锁定** | `Tag` 与锁定信息对读屏可读；避免仅用颜色区分，辅以文案 |
| **临时密码 Modal** | 标题为一级标题语义；密码区域只读输入可设 `aria-readonly="true"`；复制按钮 `aria-label="复制临时密码"` |

## 与 0.0.3 壳及现有 admin 页对齐

- **视觉**：沿用 **antd 深色 ConfigProvider**、**ProLayout** 侧栏与顶栏；内容区背景、卡片层级与 **`iterations/0.0.3/design/spec-admin-console.md` §5** token 一致。
- **信息密度**：参考 **`src/app/admin/prompts/page.tsx`**：`PageContainer` + 区内 **`Alert` / `Spin` / 表单或表格** 的间距与字号，保持 admin 模块**紧凑、可读**；本模块为**表格主导**，行高与 `size="middle"` 或 `small` 与 prompts 表单项密度**同级**，无需额外「紧凑模式」切换（PRD D1 可选项：**不做**密度切换）。
- **错误与重定向**：列表页对 **401 / 403** 的处理与 prompts 页**同一策略**，保证管理后台行为一致。
