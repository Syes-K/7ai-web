# PRD：管理后台 — 用户管理（版本 0.0.5）

## 文档信息

| 项 | 内容 |
| --- | --- |
| 版本 | `0.0.5` |
| 类型 | 管理后台业务模块 — 用户列表与运维操作 |
| 与代码库对齐 | 管理后台路由 **`/admin/**`**，源码 **`src/app/admin`**；与 **`/console`** 普通用户控制台区分。迭代 **`0.0.3`** 已交付管理后台壳与一级菜单；**「用户管理」** 当前为占位页（`src/app/admin/users/page.tsx`）。本迭代在**同一菜单与路由**下**填充真实业务能力**（列表与操作）。**用户数据模型、鉴权细节、管理 API** 由后续 **design / backend** 定义；本文档界定**管理员场景、功能边界与验收标准**。 |

## 1. 需求摘要

### 1.1 背景

系统需要由**系统管理员**在管理后台对用户账号进行日常运维：查看用户概况、调整可用状态、在忘记密码或安全事件后协助重置密码；列表可展示**登录失败锁定**等状态供运维判断（**不提供**管理端手动解锁）。迭代 `0.0.3` 已提供 **`/admin`** 布局与「用户管理」入口，但内容区仍为占位；本迭代将**替换占位**为可操作的**用户管理模块**，与既有「管理后台 vs 控制台」产品划分一致。

### 1.2 目标用户

- **主要**：已通过认证且具备**用户管理权限**的**系统管理员**（具体角色名与权限粒度由后续安全/后端方案定义；本需求要求**非管理员不可访问**该模块业务能力）。

### 1.3 核心场景

1. 管理员进入 **`/admin/users`（用户管理）**，浏览**用户列表**，掌握关键标识与当前状态（含是否处于登录失败锁定等，字段集合由设计/数据模型定；**只读展示**）。
2. 管理员对指定用户执行**状态变更**（例如启用/停用或等价状态枚举），用于风控、合规或账号治理。
3. 管理员为指定用户执行**密码重置**，使用户能用新凭据登录（具体交付方式见「待确认」）。

### 1.4 成功指标（可验收）

| 指标 | 说明 |
| --- | --- |
| 列表可用 | 在授权前提下，用户管理页能加载并展示**分页用户列表**；支持**基础刷新**；空列表、加载中、加载失败均有可理解反馈。 |
| 操作闭环 | **状态变更**、**重置密码**均可从界面发起，成功或失败有明确提示；高风险操作需**二次确认**（见用户故事 AC）。 |
| 安全与边界 | 无权限用户**不能**通过 UI 完成上述操作；敏感操作有**审计友好**的交互（如确认文案、操作对象标识）；**不**在本文档范围内承诺完整审计日志产品（若已有基础设施可对接，见依赖）。 |
| 产品一致性 | 能力挂载在既有 **`/admin`** 信息架构下，与 **`/console`** 用户自助能力**不混用**同一套文案与入口。 |

## 2. 功能范围

### 2.1 做（In Scope）

- **用户列表**：展示用户集合（字段与排序、筛选范围由设计/backend 定；PRD 要求至少能区分**身份标识**、**状态**、**锁定相关状态**等运维所需信息）。
- **用户状态变更**：在允许的状态集合内变更用户状态（具体枚举与业务含义由数据模型定义）。
- **重置密码**：触发针对单用户的密码重置流程（策略见「待确认」）。
- **权限与错误处理**：无权限、资源不存在、冲突、网络错误等均有用户可理解的反馈；破坏性操作前**确认**。

### 2.2 不做（Out of Scope）

- **普通用户控制台（`/console`）** 内的账号管理增强（本迭代仅 **admin 侧**）。
- **管理端「解锁」**：不提供解除**登录失败锁定**的管理员操作（锁定随策略过期或用户成功登录等既有机制处理）。
- **用户注册审批流、批量导入/导出、复杂高级筛选**（除非后续同一迭代扩容，默认不做）。
- **细粒度 RBAC 配置界面**（角色/权限矩阵编辑器）；若当前工程仅有「是否管理员」等简单模型，本需求**不**强制新增完整 RBAC UI。
- **完整审计日志产品**（列表筛选、导出、合规报表）；若后端已有操作日志管道，可作为**依赖**对接展示，但不作为本 PRD 必交付项。
- **设计/实现细节**：组件选型、接口路径、数据库字段 — 由 **design / backend** 产出。

### 2.3 依赖与假设

| 类型 | 内容 |
| --- | --- |
| 依赖 | 迭代 **`0.0.3`** 管理后台壳与「用户管理」菜单、路由已存在；本迭代在 **`src/app/admin/users`** 上演进。 |
| 依赖 | 存在**已认证的管理员会话**及后端对**管理类 API** 的鉴权；具体机制与 `0.0.2` 会话体系对齐（以工程为准）。 |
| 假设 | **用户**实体在服务端已存在或将在本版本 backend 中定义；列表分页、状态、锁定等字段与 **design/backend** 一致即可验收。 |
| 假设 | 密码重置与登录失败锁的**服务端策略**（锁定时长等）由 backend/登录链路定义；列表仅**展示**锁定相关派生信息时不提供解锁操作。 |

### 2.4 风险

| 风险 | 说明与缓解 |
| --- | --- |
| 误操作 | 状态变更、重置密码影响账号可用性 — 须**强确认**与清晰对象标识（设计项）；关键文案避免歧义。 |
| 密码泄露 | 若采用「展示一次性密码」类方案，须配合**一次性展示**、**复制后失效提示**等（由设计/backend 定）；见「待确认」。 |
| 权限绕过 | 仅前端隐藏按钮**不足** — 须依赖 **API 鉴权**；验收时以**接口拒绝**为准。 |

## 3. 用户故事与验收标准

### 3.1 用户列表

**作为** 系统管理员  
**我想要** 在用户管理页查看分页的用户列表及关键状态  
**以便** 快速定位目标账号并判断是否需要运维操作  

**验收标准：**

- [ ] **AC1**：已授权管理员访问用户管理路由时，页面展示**用户列表**（表格或等价结构），数据来源于后端（具体 API 由 backend 定）。
- [ ] **AC2**：列表支持**分页**（或「加载更多」等等价分页体验）；切换页码或分页大小后数据与当前页信息一致、无错乱。
- [ ] **AC3**：列表展示的信息足以支撑后续操作（至少包含**可区分用户的标识**、**用户状态**、**锁定相关展示**；字段以 design/backend 为准）。
- [ ] **AC4**：**加载中**有明确状态；**无数据**有空态说明；**请求失败**有错误提示并可**重试**或刷新。
- [ ] **AC5**：未授权用户访问时**不得**看到可操作的数据列表（以路由守卫 + API 行为为准；界面可跳转登录或提示无权限）。

---

### 3.2 用户状态变更

**作为** 系统管理员  
**我想要** 变更指定用户的状态（如启用/停用）  
**以便** 按运营或风控要求控制账号可用性  

**验收标准：**

- [ ] **AC1**：列表或详情操作中提供**状态变更**入口；仅对**允许变更**的状态显示可用操作（禁用规则由 backend 定义时前端需对齐）。
- [ ] **AC2**：执行前展示**确认**（文案含目标用户标识）；用户取消则**不**发起请求。
- [ ] **AC3**：成功后列表或当前行**反映新状态**（刷新或乐观更新，以实现为准）；失败展示**明确错误原因**（含 backend 返回信息）。
- [ ] **AC4**：无权限时**不可**完成变更（按钮不可见或禁用 + API 拒绝）。

---

### 3.3 重置用户密码

**作为** 系统管理员  
**我想要** 为指定用户触发密码重置  
**以便** 协助用户恢复登录或应对安全事件  

**验收标准：**

- [ ] **AC1**：提供**重置密码**操作入口（单用户粒度）；操作前**二次确认**（含目标用户标识）。
- [ ] **AC2**：成功后的反馈与**凭据交付方式**符合「待确认」中**最终选定策略**（例如：仅提示已发送、或一次性展示临时密码等）。
- [ ] **AC3**：失败时展示可理解错误（用户不存在、策略不允许、频率限制等 — 以后端为准）。
- [ ] **AC4**：无权限用户无法完成重置；敏感信息**不**长期停留在页面上（若有一次性密码展示，需有安全提示，细节由设计定）。

## 4. 待设计项（交接给设计）

以下需 **design** 输出交互与视觉，本 PRD 仅列清单：

| 序号 | 项 | 说明 |
| --- | --- | --- |
| D1 | **列表页布局** | 表格列宽、首屏信息优先级、是否需紧凑/密度切换（可选）。 |
| D2 | **筛选与排序** | 若 MVP 仅分页无筛选，需明确空状态与后续扩展位；若有简单搜索/筛选，字段与行为由设计列明。 |
| D3 | **行内操作 vs 批量** | 本 PRD 以**单用户操作为主**；若设计增加批量，须另附范围说明（默认不做批量）。 |
| D4 | **确认弹窗** | 状态变更、重置密码两类确认的标题、正文、主按钮文案与危险操作视觉区分。 |
| D5 | **错误与空态** | 网络错误、403、404（用户不存在）、列表为空的插图或文案层级。 |
| D6 | **密码重置成功态** | 与「待确认」策略一致：邮件发送成功提示 / 一次性密码展示 / 剪贴板与倒计时等（若适用）。 |
| D7 | **移动端/窄屏** | 管理后台默认桌面优先；最小可用宽度与表格横向滚动或卡片化由设计定（可选）。 |

## 5. 待确认项（产品侧建议）

### 5.1 密码重置策略（二选一或组合）

| 方案 | 概要 | 优点 | 缺点 |
| --- | --- | --- | --- |
| **A. 服务端生成随机临时密码，界面一次性展示** | 重置后立即展示一次临时密码；用户首次登录需改密（若项目支持） | 运维直达、不依赖邮箱 | 泄露风险高，需严格一次性展示与审计 |
| **B. 发送重置邮件/链接（若系统已有邮件能力）** | 管理员触发后用户收邮件自助完成 | 更安全、符合常见 B2B 习惯 | 依赖邮件基础设施与用户邮箱有效 |
| **C. 固定初始密码（如统一默认密码 + 强制改密）** | 运维告知用户默认密码 | 实现简单 | 安全风险大，**不推荐**作默认方案 |

**产品推荐默认**：优先 **B（邮件/链接）** 若工程已具备或短期可接入；否则采用 **A（随机临时密码 + 严格一次性展示 + 提示尽快修改）**。**不推荐**将 **C** 作为默认生产策略。

**请决策**：在 `0.0.5` 的 design/backend 阶段**锁定一种为主路径**；若需「可配置」或「A+B 降级」，在 backend 文档中写明规则与风险。

### 5.2 其他可选确认

- 「用户状态」枚举与业务名称（启用/停用/冻结等）是否与现有用户表一致。
- 管理员是否**禁止**对自己执行危险操作（如重置自己的密码由本人自助完成）；建议在 backend 层约束。

### 5.3 操作者记录（补充约定）

凡在**用户管理**（`/admin/users` 对应的管理端 API）中发起的、导致**目标用户信息或密码发生变更**的操作（含状态变更、重置密码等写操作），在需要落库或打日志记录「操作者」时，**使用当前已登录管理员的邮箱**（与会话中通过 `ADMIN_USER` 白名单校验通过的管理员身份一致）作为操作者标识，以便区分**具体经办人**；变更来源仍可通过路由/模块名（管理端用户管理）与**用户自助**渠道区分。
