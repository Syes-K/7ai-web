# 风险与开放项：MCP × 知识库 × 对话（version 0.1.9）

供阶段 3B/产品与运维对齐；部分条目在产品 `README.md` 开放问题中已有倾向，此处从**服务端**视角补充。

---

## 1. 安全

| 风险 | 说明 | 缓解方向 |
| --- | --- | --- |
| **SSRF / 内网探测** | 用户配置的 `endpoint` 可能指向 metadata 服务、内网 IP | HTTP(S) transport 做 URL 解析与黑名单/白名单、禁 loopback、限制重定向链；超时 + 最大读取字节 |
| **凭证泄露** | DB 备份、日志、错误栈 | 应用层加密存储、日志脱敏、测试连接接口不返回堆栈 |
| **子进程命令注入** | 若 `stdio` transport 将用户输入拼进 shell | **禁止** `shell: true`；使用参数数组 exec；校验可执行路径与白名单 |
| **DoS** | 大量 MCP 或慢 `list_tools` 阻塞对话 | 每 turn 最大绑定数、单 MCP 超时、`Promise.allSettled`、必要时取消 token |

---

## 2. 性能与成本

| 风险 | 说明 |
| --- | --- |
| **每轮建连** | 若每次用户消息都新建 MCP 连接 + `list_tools`，延迟与资源消耗显著 |
| **工具爆炸** | 多 MCP × 每工具多 schema → 上下文膨胀、模型选型困难 |

**缓解（供 3B 选型）**：连接池/按会话缓存 MCP 客户端（需注意失效与并发）；对 tools 数量做软上限并记录 warn；长期可做「按 KB 分组工具描述」等非本期项。

## 4. 产品与行为仍可能变更

以下在设计 `design-spec.md` 中已有**默认假设**，若产品改稿需回溯实现：

- MCP 与 RAG **解耦（提案 A）** vs 条件挂载（提案 B）。
- 删除 MCP：**禁止硬删** vs 级联解除挂载。
- 知识库保存时是否允许仅含 `enabled=false` 的挂载（当前：**允许**，运行时忽略）。

---

## 5. 可观测性与用户可见性

- **ToolTrace / onToolEvent**：PRD 提到与 `assistant.ts` 方向一致；需确认现有流式链路是否已覆盖「工具注册失败」类事件，避免仅 `console.error`。
- **对话 Toast**：部分 MCP 失败时「每用户消息轮次至多一次」需在后端或前端统一防抖（设计在 `spec-chat-agent-mcp.md`）；若仅后端检测，需约定 SSE/响应头或侧信道传递轻量信号（**开放**：是否首期仅前端根据工具数量变化推断，不推荐）。
