## 1. 需求摘要

### 1.1 背景

控制台已提供「模型管理」入口（见 `src/app/console/console-menu.tsx`），但页面仍为占位（`src/app/console/models/page.tsx` 使用 `ProModulePlaceholder`）。业务侧需要在控制台完成**可配置的模型清单**管理：为后续对话/助手等能力提供可选的 LLM 实例配置。服务端已有与厂商兼容的 Base URL 映射（`MODEL_PROVIDER_BASE_URL`），本迭代聚焦控制台侧**列表展示与增改能力**及与上述键的一致性。

### 1.2 目标用户

- **控制台登录用户**（当前产品未在本 PRD 中拆分多角色；若存在管理员/普通用户差异，见「待产品确认的问题」）。

### 1.3 核心场景

1. 用户进入「模型管理」，查看已配置的模型列表（字段至少包含：模型名称、Provider、与密钥相关的展示策略由设计/实现约定）。
2. 用户新增一条模型配置：选择 Provider（限定为已支持键）、填写模型名称、填写该模型使用的 API Key。
3. 用户修改已有配置：可更新模型名称、Provider、API Key（若产品确认允许变更 Provider，见待确认项）。
4. 用户删除一条不再需要的模型配置：经确认后从列表移除，服务端**物理删除**该条记录（与 `iterations/0.0.8/backend/api-spec.md` 一致）。

### 1.4 成功指标（可量化）

| 指标 | 说明 |
|------|------|
| **配置闭环** | 用户可在控制台完成「新增 → 列表可见 → 编辑保存 → **删除**」全流程，无阻塞性缺陷（以验收标准为准）。 |
| **一致性** | 所有已保存模型的 `provider` 值仅能为 `ALYUN`、`GLM`、`DEEPSEEK` 之一（与代码常量一致，大小写敏感）。 |
| **可理解性** | Provider 在界面上展示**人类可读名称**（如阿里云百炼、智谱、深度求索），与底层键的映射关系在实现中固定且可追溯。 |

## 2. 功能范围

### 2.1 本期要做（In Scope）

| # | 能力 | 说明 |
|---|------|------|
| 1 | **模型列表** | 使用 **ProTable**（`@ant-design/pro-components`）展示已配置模型；列至少包含：模型名称、Provider（展示名）、以及密钥相关列的策略（见待确认）。支持表格常见能力：**分页**、**刷新**；若数据量预期较小，可采用前端分页或简单服务端分页（由设计与后端约定）。 |
| 2 | **新增模型** | 表单或抽屉/弹窗：选择 Provider（三选一，键与代码一致）、录入模型名称、录入 API Key。提交后列表反映新数据。 |
| 3 | **修改模型** | 对已有记录进入编辑态，可更新模型名称、Provider、API Key（具体是否允许改 Provider 见待确认）。 |
| 4 | **Provider 枚举** | 仅允许以下**字符串键**（与 `MODEL_PROVIDER_BASE_URL` 一致）：`ALYUN`（阿里云百炼）、`GLM`（智谱）、`DEEPSEEK`（深度求索）。不允许自由文本输入 provider。 |
| 5 | **删除模型** | 在列表中对单条记录提供删除能力；须二次确认；成功后该配置不再出现在列表中；**物理删除**（无单独「停用」态，除非后续迭代扩展）。 |

### 2.2 本期不做（Out of Scope）

- 对话页/助手页**实际调用**时如何选用列表中的哪一条模型（路由策略、默认模型）——除非已有全局约定，否则不在本 PRD 展开。
- Provider 扩展（新增第四家厂商）或修改 Base URL（属服务端配置/发版范畴）。
- API Key 的**轮换策略**、**用量计费**、**模型能力探测**（如列出远端可用 model id）。
- 与「助手管理」「知识库」等模块的联动配置（可后续迭代衔接）。

### 2.3 依赖与假设

| 类型 | 内容 |
|------|------|
| **依赖** | 需要**持久化存储**与 **API**（创建/查询/更新/**删除**）以支撑列表与表单；具体数据模型与接口由后续后端/设计文档定义。当前 `getModel` 使用 `process.env.CHAT_LLM_API_KEY` 的写法与「按模型录入 API Key」并存时，**运行时如何选用密钥**需在后端阶段明确（本 PRD 仅要求控制台具备录入与展示能力）。 |
| **假设** | 单用户或单租户场景下「模型配置」对当前登录用户可见；若需按组织隔离，待确认。 |
| **现状** | `src/app/console/models/page.tsx` 为占位；本迭代需求落地后由实现替换为真实页面（实现细节不属本文档）。 |

## 3. 用户故事与验收标准（AC）

### US-1：查看模型列表（ProTable）

**作为** 控制台用户  
**我想要** 在「模型管理」页用表格查看已配置的模型  
**以便** 了解当前可用的模型与 Provider 分布  

**验收标准：**

- [ ] AC1：页面使用 ProTable 展示列表，至少包含列：**模型名称**、**Provider**（展示为中文可读名称，与三枚键对应）。
- [ ] AC2：列表数据来自服务端（或约定数据源），加载中有明确反馈；加载失败有错误提示。
- [ ] AC3：支持分页与刷新（刷新后数据与后端一致）。
- [ ] AC4：Provider 列不得出现 `MODEL_PROVIDER_BASE_URL` 未定义的键。

---

### US-2：新增模型

**作为** 控制台用户  
**我想要** 新增一条模型配置（含 Provider、模型名称、API Key）  
**以便** 在系统中登记可用的 LLM 接入信息  

**验收标准：**

- [ ] AC1：提供入口（如「新建」按钮）打开表单；**Provider** 仅为三选一：`ALYUN` / `GLM` / `DEEPSEEK`（提交到后端的值与上述键一致）。
- [ ] AC2：**模型名称**为必填；**API Key** 为必填（是否允许空格 trim、最小长度等由实现校验，但需有基本非空校验）。
- [ ] AC3：提交成功后在列表中可见新记录；失败时提示原因。
- [ ] AC4：同一表单内对非法 Provider 无提交路径（无自定义 provider 字符串）。

---

### US-3：修改模型

**作为** 控制台用户  
**我想要** 编辑已有模型配置  
**以便** 更正名称、更换 Provider 或更新 API Key  

**验收标准：**

- [ ] AC1：每条记录可进入编辑（行内编辑或弹窗/抽屉，由设计定）。
- [ ] AC2：可更新**模型名称**与 **API Key**；若产品确认允许修改 **Provider**，则编辑表单与新增一致为三选一。
- [ ] AC3：保存成功后列表反映最新数据；并发或版本冲突有合理提示（若后端提供）。

---

### US-4：删除模型

**作为** 控制台用户  
**我想要** 删除一条不再使用的模型配置  
**以便** 保持列表整洁并避免误用已废弃的密钥配置  

**验收标准：**

- [ ] AC1：每条记录提供**删除**入口（与编辑并列或置于操作列），触发后须有**二次确认**（文案需明示不可恢复或与服务端删除语义一致）。
- [ ] AC2：删除成功后该记录从列表消失；若当前页删空且非第一页，行为与项目惯例一致（如回到上一页或刷新列表）。
- [ ] AC3：删除失败时提示原因；无权限或记录不存在时提示清晰（与后端约定一致，避免泄露他人资源）。

## 4. 待设计项清单（供 design 接手）

以下需交互与视觉设计定稿，本 PRD 不规定具体组件细节：

| # | 项 | 说明 |
|---|----|------|
| D1 | **页面信息架构** | `PageContainer` 标题区、工具栏（新建、刷新）、表格区的布局与间距，与控制台其他页（如助手、设置）一致。 |

## 5. 待产品确认的问题

以下问题影响范围与验收口径；**已收敛项**保留记录，**仍开放项**继续跟进：

1. **删除/停用**：**已确认：需要删除**；本期为**物理删除**（无「停用」态）。若未来需审计追溯，可再评估软删除。
2. **Provider 是否可改**：编辑时是否允许变更 `provider`（可能影响已引用该模型的其他模块）？
3. **API Key 回显**：编辑时若用户不修改 Key，是否支持「留空表示不变」？还是每次必须重新输入？
4. **权限**：是否所有控制台用户均可管理模型，还是仅特定角色？（若暂无角色体系，可默认「当前登录用户均可」。）
5. **密钥安全**：列表/API 返回是否**永不**返回完整明文 Key（仅掩码）；与合规/审计相关的日志是否在本期要求？
