# PRD：控制台「个人信息 + 用户配置」合并页（0.0.9）

## 文档信息

| 项 | 内容 |
|----|------|
| 版本 | 0.0.9 |
| 状态 | 需求阶段（供设计接手） |
| 关联迭代 | 控制台占位页升级为真实功能；与 0.0.8 已上线的「模型登记」能力衔接 |

---

## 1. 需求摘要

### 1.1 背景

控制台当前存在「个人信息」（`/console/profile`）与「用户配置」（`/console/settings`）两个入口，页面均为 `ProModulePlaceholder`。用户希望在**同一页面**内用**两个模块**分别承载两类能力，降低跳转与心智负担；两类数据职责不同，需**分表管理**并与现有实体对齐。

### 1.2 目标用户

- 已登录、使用控制台的终端用户（与现有控制台受众一致）。

### 1.3 核心场景

1. **查看**：进入页面后默认以**预览/只读**态查看个人信息与用户配置，无需误触即改。
2. **编辑个人信息**：在明确操作（如「编辑」）后进入编辑态，可修改昵称、手机号等；**邮箱始终不可改**（与账号体系一致）。
3. **编辑用户配置**：在明确操作后进入编辑态，从用户已登记的模型配置（`UserModelConfig` 列表）中**选择一条**作为系统应使用的「默认/偏好」语义（具体语义见「待确认问题」）；选项仅包含该用户名下已存在的登记项。

### 1.4 成功指标（建议）

- 用户可在单页完成「看资料 + 看/改默认模型偏好」，无需在两个占位页之间切换。
- 个人信息编辑与配置编辑均有明确进入/退出编辑态的路径，只读态下无意外提交。
- 与数据模型一致：个人信息变更落在用户主表；「选中哪条模型配置」不混入 `user_model_configs` 行内业务含义混乱（见第 5 节产品假设）。

## 2. 功能范围

### 2.1 本期要做（In Scope）

| 域 | 说明 |
|----|------|
| 单页双模块 | 一个页面内两个模块：**个人信息**、**用户配置**；布局与视觉由设计定义（见第 6 节）。 |
| 预览优先 | 默认**只读/预览**；通过用户操作（如按钮）进入**编辑模式**；保存或取消后回到只读（交互细节由设计定）。 |
| 个人信息 | 展示并可编辑（在允许字段内）：**昵称**、**手机号**（与现有 `User` 字段一致）；**邮箱展示但不可编辑**。 |
| 用户配置 | 提供「**从已登记模型中选一条**」的能力，与 `UserModelConfig`（`user_model_configs` 表）关联；需在界面与文案上表达为「默认使用的模型配置」或等价语义（待确认精确产品语言）。 |
| 路由/菜单策略 | 在 PRD 层提出选项，由产品与 design 确认：保留双菜单指向同一页、或合并为一项（见第 6 节）。 |

### 2.2 本期不做（Out of Scope）

- **模型登记的 CRUD**：新增/编辑/删除 API Key 与模型条目仍以现有「模型管理」流程为准；本页**仅消费**已存在列表。
- **修改邮箱**、**改密**、**账号注销**等账号安全大类（除非已有独立 Story，否则不纳入本 PRD）。
- **多设备同步、审计日志、字段级变更历史**（除非明确列为后续迭代）。
- **未登记任何模型时**的完整引导流程：可定义最小行为（如禁用选择器 + 提示去模型管理），复杂空态营销化交由后续迭代。

### 2.3 依赖与假设

- 依赖现有登录与会话；依赖 `users` 与 `user_model_configs` 数据模型及 0.0.8 控制台模型清单能力。
- 假设「选中模型配置」需持久化，且与「某条 `UserModelConfig` 记录」是**多对一**关系（每用户最多一个「当前选中」指针）；具体落库方式见第 5 节，由后端在接口/表结构方案中实现。

## 3. 用户故事与验收标准

### US-1：单页查看个人信息与用户配置（只读）

**作为** 控制台用户  
**我想要** 在同一页面以只读方式查看我的昵称、邮箱、手机号以及当前选中的模型配置  
**以便** 快速确认资料与默认模型偏好，且不会误修改  

**验收标准：**

- [ ] 进入页面后默认为**只读/预览**态，个人信息与配置模块均**不**处于可编辑输入状态（或等价交互：无未确认变更）。
- [ ] **邮箱**以只读形式展示，**不提供**改为其他邮箱的入口。
- [ ] 展示**昵称**、**手机号**（若为空则展示约定空态文案，由设计定）。
- [ ] **用户配置**模块展示「当前选中的模型配置」的可识别信息（至少能区分于其他登记项，如 provider + modelName 或列表中的标题规则，与现有控制台展示一致）。
- [ ] 若用户尚未登记任何 `UserModelConfig`，配置模块有**明确空态**（文案与是否引导至模型管理页由设计定，见「待确认问题」）。

### US-2：编辑个人信息

**作为** 控制台用户  
**我想要** 通过明确操作进入编辑态并修改昵称、手机号  
**以便** 保持联系方式与展示名称准确  

**验收标准：**

- [ ] 仅通过**显式用户操作**（如「编辑」）进入个人信息编辑态；取消或保存后回到只读态。
- [ ] **邮箱**在编辑态下仍为**不可编辑**（禁用或只读展示）。
- [ ] **昵称**、**手机号**可编辑；校验规则与后端一致（如手机号格式、唯一性、可空策略以现有 `User` 约束为准）。
- [ ] 保存成功后在只读态反映最新值；失败时提示可理解错误原因。

### US-3：编辑用户配置（选择已登记模型中的一条）

**作为** 控制台用户  
**我想要** 在已登记的模型配置中选择一条作为默认/偏好  
**以便** 对话或其它功能默认使用我指定的接入配置  

**验收标准：**

- [ ] 仅通过**显式用户操作**进入用户配置编辑态。
- [ ] 选择器/列表**仅包含**当前用户名下已存在的 `UserModelConfig` 记录（与模型管理页数据源一致）。
- [ ] 保存后，只读态展示与所选记录一致；刷新页面后仍保持一致（持久化成功）。
- [ ] **不得**在本模块内要求用户重新输入 API Key；密钥管理仍在模型管理流程中。

### US-4：导航与入口（与范围一致）

**作为** 控制台用户  
**我想要** 从侧边栏进入该页面  
**以便** 管理资料与配置  

**验收标准：**

- [ ] 至少保留**一个**控制台菜单入口可到达本页（实现方式：单菜单项或双项同页，见第 6 节待设计项）。
- [ ] 若保留 `/console/profile` 与 `/console/settings` 两个 URL，两者打开**同一页面内容**或**明确主次**（由设计定），避免两套不一致内容。

## 4. 与代码现状的对齐说明（仅需求引用，非实现说明）

- **`User`（`users` 表）**：`email`（唯一）、`telNo`（可空、唯一）、`nickName`、`passwordHash` 等；本 PRD 个人信息模块字段以该实体为准。
- **`UserModelConfig`（`user_model_configs` 表）**：用户维度模型接入配置（provider、modelName、apiKeyCipher 等）；本 PRD「可选模型列表」即该用户下多条登记记录。
- **控制台**：`/console/profile`、`/console/settings` 当前为占位；菜单定义见 `console-menu`（需求层知悉即可）。

---

## 5. 两模块与数据表的对应关系（产品假设，供后端细化）

| 模块 | 主要业务数据 | 表 / 实体（现状） | 产品假设 |
|------|----------------|-------------------|----------|
| 个人信息 | 邮箱、手机、昵称等 | `users` / `User` | 读写仍主要落在 **`users`**；邮箱只读为产品规则，非新表。 |
| 用户配置（选中模型） | 「当前使用哪一条已登记配置」 | `user_model_configs` / `UserModelConfig` 提供**可选列表** | **列表来源**为已有表；**「当前选中哪一条」**在现状实体中**无字段**，产品假设需新增以下之一（后端选型）：<br>• **方案 A**：在 **`users` 表**增加可空外键/指针字段，如 `preferredModelConfigId` → 指向该用户某条 `UserModelConfig.id`；<br>• **方案 B**：新建轻量表（如 `user_preferences`）存 `userId` + `selectedModelConfigId`；<br>• **方案 C**：若产品确认「偏好」与账号强绑定且仅一项，仍可能落在 `users` 单列。 |

**约束（产品侧）：**

- 「选中」必须引用**已存在且属于当前用户**的 `UserModelConfig.id`；删除某条登记时，需定义行为（自动清空选中 / 提示不可删 / 级联规则），由后端与产品共同在接口层明确。
- 不在本 PRD 中规定加密字段细节；前端本模块不展示完整密钥。

## 6. 待设计项（Design Handoff）

以下需 **design** 出稿或交互说明，本 PRD 不锁定像素级方案。

1. **单页双模块布局**：上下分区或左右分区；模块标题与层级（是否与页面总标题合并）。
2. **预览/编辑切换**：每模块独立「编辑/保存/取消」还是页面级统一编辑；未保存离开是否拦截。
3. **用户配置选择控件**：下拉、单选列表或卡片；只读态如何摘要展示（provider + modelName 等）。
4. **路由与菜单**：**选项 A** 保留「个人信息」「用户配置」两项菜单，均指向同一 URL（或其一重定向）；**选项 B** 合并为一项（如「账号与偏好」）；**选项 C** 单一路由，废弃另一路径 301/302。需标注推荐项与 SEO/书签影响（若有）。
5. **空态与错误态**：无模型登记、保存失败、网络错误、选中项已被删除后的降级展示。
6. **移动端/窄屏**（若控制台需支持）：模块堆叠与编辑面板行为。

## 7. 待确认问题（需产品/业务拍板）

1. **「可用模型」的语义**：所选的一条 `UserModelConfig` 是 **(a)** 全站/全局默认（所有对话新会话默认用该配置）、**(b)** 仅控制台内默认、还是 **(c)** 仅作为展示偏好不影响实际调用？这决定文案（「默认模型」「偏好模型」）与后续消费方（对话模块）的依赖说明。
2. **未登记模型时**：是否强制引导跳转「模型管理」；是否允许页面其余部分仍可用。
3. **手机号**：是否必填、修改时是否要短信验证（本期是否仅后端校验格式+唯一性）。
4. **国际化**：本期是否仅中文文案。
5. **与 `/console/models` 的联动**：保存偏好后，是否在模型管理页显示「当前默认」角标（本期是否做）。

## 8. 优先级建议（供排期参考）

1. P0：只读展示 + 个人信息编辑 + 模型偏好选择与持久化（含空态最小方案）。
2. P1：路由/菜单合并或统一、删除登记项时的选中降级策略的产品文案与流程。
3. P2：审计、变更历史、高级账号安全。
