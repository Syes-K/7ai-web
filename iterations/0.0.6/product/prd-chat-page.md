# PRD：对话页 — AI 聊天与会话管理（版本 0.0.6）

## 文档信息

| 项 | 内容 |
| --- | --- |
| 版本 | `0.0.6` |
| 类型 | 普通用户侧 — 对话页（AI 聊天场景） |
| 与代码库对齐 | 路由 **`/chat`**（`src/app/chat`）当前为占位；本迭代界定**会话创建、持久化、历史切换、清空消息**等产品能力。具体布局、组件形态、接口契约由 **design / backend** 定义；本文档不写视觉与交互稿级细节。 |

## 1. 需求摘要

### 1.1 背景

用户需在**已登录**前提下使用 AI 对话能力完成咨询、创作与多轮协作。仅「单页即时聊」不足以支撑长期使用：用户会希望**开启新话题**、**离开后再回来继续**，以及**清空当前话题内容**而不必删除整条会话记录。工程上已有对话页入口与布局骨架，本版本将占位升级为**可持久化的多会话体验**。

### 1.2 目标用户

- **主要**：已通过认证、使用 **`/chat`** 的**终端用户**（与普通用户控制台同一身份体系；非管理后台场景）。

### 1.3 核心场景

1. 用户进入对话页，可**新建会话**并开始与模型多轮交互。
2. 用户再次访问时，可从**历史会话列表**中选择某条记录，**加载该会话的消息**并继续发送消息。
3. 用户可对**当前正在查看的会话**执行**清空**：删除该会话下的**消息内容**（见下文「清空」定义），**保留**该会话在历史中的**条目**（标题/时间等元数据可更新规则由设计/backend 定，PRD 要求至少「条目仍在列表中可识别」）。
4. （**本版本不做，见范围**）从产品中**永久删除某条历史会话**（整条记录移除）— 标为可选扩展，默认不交付。

### 1.4 成功指标（可验收）

| 指标 | 说明 |
| --- | --- |
| 新会话可用 | 用户可明确发起**新会话**，新会话与既有历史会话**互不混淆**（切换后上下文独立）。 |
| 持久与恢复 | 会话及消息**持久化**；刷新页面、重新登录（在会话有效前提下）后仍能**从历史进入并看到已存消息**并继续对话。 |
| 历史可导航 | 用户可浏览历史列表并**切换**到任意已存会话；当前选中会话在界面上可识别（具体表现由设计定）。 |
| 清空语义一致 | 「清空」仅影响**当前会话的消息**；该会话仍在历史中可见；用户不会因清空而「丢失会话条目」或无法新建/切换会话。 |
| 边界清晰 | 未登录用户行为与既有 **`/chat`** 鉴权策略一致（如重定向登录）；无权限场景不暴露管理类能力。 |

## 2. 功能范围

### 2.1 做（In Scope）

- **新建会话**：在对话页提供可发现的入口，创建一条新的会话记录，并进入该会话的「空消息」状态（或等价初始状态），用户可立即输入首条消息。
- **消息收发与展示**：在当前会话内展示多轮用户/助手消息；发送新消息并展示模型回复（具体模型与流式与否由 backend/既有链路定，本 PRD 要求**可完成一轮以上闭环**）。
- **会话持久化**：会话元数据与消息内容**服务端持久化**，按**当前登录用户**隔离；不得将 A 用户会话暴露给 B 用户。
- **历史列表**：展示当前用户的历史会话列表（排序、分页或「加载更多」、默认展示条数由 design/backend 定；PRD 要求**可用且可扩展**，避免仅内存态列表）。
- **切换会话**：从历史选择一项后，主区域展示该会话的消息；切换前若有未保存草稿类状态，行为由设计定义「丢弃或提示」— PRD 最低要求：**切换后消息与当前会话一致，无串会话**。
- **清空当前会话消息**：清除**当前选中会话**下的全部消息（或等价「消息列表为空」）；**保留**该会话在历史列表中的**条目**（可显示「空会话」或更新标题/时间，由设计定）。需**确认操作**以防误触（文案与形式由设计定）。

### 2.2 不做（Out of Scope）

- **删除整条历史会话**（从历史列表永久移除该会话记录）：**本版本不交付**；若未来需要，可作为独立故事/版本追加（含回收站、批量删除等不在此展开）。
- **跨用户/分享会话链接、导出对话、搜索全库消息**等高级能力。
- **管理后台**对会话的审计、运营查看（除非与既有安全要求强制重叠，默认不在此 PRD）。
- **视觉稿、组件库选型、动效参数、具体 API 路径与数据库表结构** — 由 **design / backend** 产出。

### 2.3 依赖与假设

| 类型 | 内容 |
| --- | --- |
| 依赖 | 既有 **`/chat`** 路由与布局、middleware 对登录态的约束；本迭代在 **`src/app/chat`** 及相关服务端能力上演进。 |
| 依赖 | 存在**已认证用户会话**；用户标识可用于**数据隔离**（与 `0.0.2` 等既有鉴权对齐，以工程为准）。 |
| 依赖 | AI 调用链（如 `src/server/llm` 等）需能接收**会话/消息上下文**以便多轮连续；具体传参与摘要策略由 backend 定。 |
| 假设 | 「会话」有稳定 **ID**；消息有顺序与角色（用户/助手等）；列表与详情查询在性能上可接受（分页/索引由 backend 定）。 |
| 假设 | 首条消息前**会话标题**可采用首条用户消息截断、默认「新对话」等策略之一（由 design/backend 二选一或组合，需在开放问题中收敛）。 |

### 2.4 风险

| 风险 | 说明与缓解 |
| --- | --- |
| 误清空 | 清空不可依赖「单击即生效」— 须**二次确认**；文案明确「仅清空消息、会话仍在列表中」。 |
| 数据隔离 | 验收须包含**接口与列表**均不可越权访问他人会话 ID。 |
| 长会话性能 | 消息量极大时的加载与 token 策略由 backend/LLM 层处理；前端至少不因「无分页」而一次性卡死（可与设计约定列表与消息区加载策略）。 |

## 3. 用户故事与验收标准

### 3.1 新建会话

**作为** 已登录用户  
**我想要** 在对话页开启一条新的会话  
**以便** 与当前历史话题分开，从零开始讨论新主题  

验收标准：

- [ ] AC1：存在明确入口（如按钮或等价操作）可**新建会话**；操作后进入**新会话上下文**，主消息区不展示**其他会话**的消息。
- [ ] AC2：新建后可立即输入并发送首条消息；发送成功后消息归属于**该新会话**（刷新或重新进入历史后仍可看到）。
- [ ] AC3：新建会话后，该会话出现在**历史列表**中（可与发送首条消息后写入等实现策略一致，但用户可感知「已有一条可回访的会话」）。

---

### 3.2 会话持久化与历史列表

**作为** 已登录用户  
**我想要** 我的对话被保存，并能在历史中看到  
**以便** 稍后继续而不用从头开始  

验收标准：

- [ ] AC1：**刷新页面**后，此前已产生的会话仍出现在历史列表中，且消息不丢失（在服务端已成功持久化的前提下）。
- [ ] AC2：历史列表仅展示**当前用户**的会话，不展示其他用户数据。
- [ ] AC3：列表中单条会话至少具备**可区分信息**（如标题/时间/预览其一或组合，具体字段由设计定）；空消息会话仍可出现在列表中（与「新建」「清空」行为一致）。

---

### 3.3 切换会话并继续对话

**作为** 已登录用户  
**我想要** 从历史中选择某次对话并继续发送消息  
**以便** 在同一页面延续之前的讨论  

验收标准：

- [ ] AC1：点击（或等价选择）历史中的某条会话后，主区域展示**该会话的全部已持久化消息**（顺序正确）。
- [ ] AC2：在该状态下发送的新消息归属于**当前选中会话**，并出现在消息列表末尾；切换至其他会话后再切回，仍能看到这些消息。
- [ ] AC3：切换会话时**不出现**会话 A 的消息混入会话 B 的展示（验收可做：交替切换两个有内容的会话，内容始终正确）。

---

### 3.4 清空当前会话消息

**作为** 已登录用户  
**我想要** 清空当前会话中的消息  
**以便** 在同一「会话条目」下重新开始讨论，而不必新建一条历史记录  

验收标准：

- [ ] AC1：「清空」指：**删除当前会话下的全部消息记录**（用户与助手消息均不再展示）；操作后主消息区为**空**或等价初始态。
- [ ] AC2：清空后，该会话**仍保留在历史列表中**（条目不消失）；用户仍可选中该会话并发送新消息（形成新的消息序列）。
- [ ] AC3：执行清空前须有**确认步骤**；确认后给出成功或失败反馈（失败时消息应保持不变或给出可理解错误原因）。
- [ ] AC4：清空**仅影响当前会话**，不影响其他历史会话的消息。

---

### 3.5 未登录与错误场景（补充）

**作为** 系统  
**我想要** 未授权用户无法访问他人数据  
**以便** 保证隐私与安全  

验收标准：

- [ ] AC1：未登录用户访问对话能力的行为与既有 **`/chat`** 策略一致（如重定向登录），不暴露历史与消息 API 的实质数据。
- [ ] AC2：对非法会话 ID、网络错误、服务端错误等，界面有**可理解反馈**，不白屏、不静默失败。

---

## 4. 待设计项清单（供 design，非设计稿）

- 对话页信息架构：**历史列表**与**主对话区**的布局关系（侧栏/抽屉/底部等）。
- 新建会话、清空、当前会话选中态的**可见性与防误触**（含清空确认样式）。
- 历史列表项的**标题规则**、时间格式、空会话/已清空会话的展示文案。
- 加载中、空列表、无网络、长列表的**状态与反馈**。
- 若存在**输入草稿**：切换会话时是否保留、如何提示（与 PRD 最低要求对齐即可）。

## 5. 开放问题

| # | 问题 | 说明 |
| --- | --- | --- |
| O1 | 会话标题生成规则 | 默认「新对话」、首条用户消息截断、或用户可编辑 — 需与 design/backend 收敛一种或分阶段。 |
| O2 | 单用户会话数量上限 | 是否限制列表长度、是否归档 — 若不限，需 backend 考虑分页与性能。 |
| O3 | 清空后模型侧上下文 | 若服务端使用摘要/向量上下文，清空消息是否**同步清除**派生状态 — 由 backend 保证与「消息已删」一致，避免模型「记得」已清空内容。 |
| O4 | 「删除整条历史」是否列入路线图 | 当前 **Out of Scope**；若产品后续需要，单独开需求与数据删除合规评估。 |
