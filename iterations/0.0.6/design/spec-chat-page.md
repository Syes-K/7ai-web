## 1. 信息架构与页面结构

### 1.1 顶层分区

页面在**已登录**前提下由两大逻辑区域组成：

| 区域 | 职责 |
| --- | --- |
| **历史区（会话列表）** | 展示当前用户的历史会话、支持选中、新建入口、清空当前会话入口（或集中在工具栏）。 |
| **主对话区** | 展示当前选中会话的消息时间线、输入区、发送与流式/加载反馈。 |

两区在**数据与交互上从属关系**为：历史区决定「当前 `conversationId`」；主对话区始终只渲染**当前选中会话**的消息与输入，**禁止**与其他会话混显（对应 PRD 切换验收）。

### 1.2 布局建议（响应式）

| 断点 | 布局 | 说明 |
| --- | --- | --- |
| **≥ 992px（`lg` 及以上，桌面）** | **左侧固定宽度侧栏（历史区） + 右侧弹性主区（对话区）** | 侧栏宽度建议 280px～320px（可收缩一档至 240px 以适配小屏桌面）。主区占满剩余宽度，输入区贴底。 |
| **768px～991px（平板竖屏 / 小屏笔记本）** | 仍可采用**窄侧栏 + 主区**，或侧栏改为**可折叠图标条**；以不挤压消息可读宽度为准。 |
| **< 768px（`md` 以下，手机）** | 历史区默认**隐藏**；通过 **顶部「历史」按钮**打开 **Drawer（推荐）**：从左侧或右侧滑入，宽度约 85vw～100vw，遮罩点击关闭。备选：**底部 Sheet** 展示列表（拇指友好，但一屏高度有限，长列表需滚动），若采用需在文档中统一「列表在 Sheet 内滚动、主区不被顶起」。 |

**断点数值**：与 antd Grid 默认断点（`xs`/`sm`/`md`/`lg`/`xl`）对齐即可，便于 `useBreakpoint` 或 CSS 媒体查询一致实现。

### 1.3 与既有代码的衔接

- 当前 `ChatPage` 使用 `PageShell` + 占位；升级时保留 **页级标题「对话」** 或按产品统一导航，**历史 + 主区**作为 `PageShell` 主内容区内嵌布局（不强制改变全站顶栏）。

## 2. 主用户流程

### 2.1 进入页面

1. 已登录用户导航至 `/chat`（或从站内入口进入）。
2. **首屏加载**：拉取历史列表（及默认选中规则，见 §4）；主对话区根据选中会话拉取消息。
3. **默认选中策略（建议）**：若有历史，默认选中**列表排序的第一条**（通常为最近活跃）；若无历史，进入「无选中 / 占位」状态，并突出 **新建会话**（见 §5.1）。

### 2.2 新建会话

1. 用户点击 **新建会话**（位置见 §5.1）。
2. 客户端进入**新会话上下文**（新 `conversationId` 由服务端创建后返回）；主消息区**清空**，不展示其他会话消息。
3. 输入区获得焦点或可立即输入；用户发送首条消息后，消息归属该会话；历史列表出现该条目（可与「创建会话」与「首条消息成功」的时序对齐，由 backend 定，界面需容忍短暂「列表刷新」加载态）。

### 2.3 发送消息

1. 用户需处于**已选中某一会话**（含新建后的会话）状态。
2. 在输入区输入 → 发送 → 消息列表追加用户消息 → 展示助手回复（加载/流式由实现定）。
3. 发送失败时：错误提示 + 可选重试；**不**将失败消息误记到其他会话。

### 2.4 切换历史会话

1. 用户在历史区点击某条会话。
2. **当前选中项高亮**（§5.2）；主区切换为该会话消息列表；输入区展示**该会话对应草稿**（§5.4）。
3. **无路由级跳转**：仍为 `/chat` 单页，可选 **URL query** 同步 `?c=` 仅作可分享/刷新恢复（若本迭代不做深链，可仅内存态）。

### 2.5 清空当前会话消息

1. 用户点击 **清空**（需防误触，见 §5.3）。
2. 弹出 **二次确认对话框**；确认后请求清空。
3. 成功：主消息区为空态；该会话仍在列表中，展示规则见 §6；**可选**成功反馈（Message 轻提示）。
4. 失败：错误说明，消息区保持原状。

## 3. 关键界面 / 组件清单

| 组件 / 模块 | 职责 |
| --- | --- |
| **会话列表容器** | 滚动列表、加载更多或分页入口、空列表占位。 |
| **历史列表项** | 单条会话：标题、时间、可选预览；选中态。 |
| **工具栏** | **新建会话**、**清空当前会话**（仅当有选中会话时可用或始终可见但禁用）、移动端 **打开历史 Drawer** 的触发器。 |
| **消息列表** | 用户/助手气泡或块、时间分隔（可选）、滚动到底部策略。 |
| **输入区** | 多行文本、发送按钮、禁用态（未登录不会出现于本页；发送中禁用重复提交）。 |
| **确认对话框（Modal）** | 清空确认；焦点陷阱与 ESC 行为见 §8。 |
| **全局反馈** | Message / notification：网络错误、清空成功/失败等。 |

**实现提示**：antd 侧可映射为 `Layout` + `Sider` / `Drawer` + `List` + `Modal` + `Input.TextArea` + `Button` 等，按设计 token 微调间距即可。

## 4. 各组件 / 界面状态

### 4.1 会话列表

| 状态 | 表现 |
| --- | --- |
| **默认** | 展示已加载的会话行；当前选中行高亮。 |
| **加载中** | 列表骨架屏（Skeleton）或 Spin 居中；避免整页白屏。 |
| **空态（无历史）** | 文案如「暂无对话记录」+ 引导按钮「新建会话」。 |
| **错误** | 行内或列表顶部 Alert：简短原因 + **重试**。 |
| **边界（长列表）** | **分页或「加载更多」**（与 PRD「可扩展」一致）；列表区独立滚动，不与主消息区滚动链死。滚动位置记忆可选（非必须）。 |

### 4.2 主消息区

| 状态 | 表现 |
| --- | --- |
| **默认** | 展示当前会话消息；输入区可用。 |
| **加载中** | 切换会话后拉取消息时：消息区 Spin 或骨架；**不**显示上一会话内容（防串会话：可先清空再加载，或覆盖加载）。 |
| **空态（当前会话无消息）** | 简短说明如「发一条消息开始对话」；与「从未发送」和「已清空」在文案上可区分（见 §6）。 |
| **发送中 / 流式** | 输入禁用或仅禁用发送；助手占位气泡或打字指示器（由实现定）。 |
| **错误** | 发送失败：气泡旁或顶部提示；加载消息失败：错误态 + 重试。 |

### 4.3 输入区

| 状态 | 表现 |
| --- | --- |
| **默认** | 可输入，发送可用（非空校验）。 |
| **发送中** | 防重复提交。 |
| **无选中会话** | 若产品允许「必须先新建」，则禁用并提示；若默认自动新建，则无此态。 |

### 4.4 清空流程

| 状态 | 表现 |
| --- | --- |
| **确认前** | Modal 展示风险说明（§5.3）。 |
| **提交中** | 确认按钮 loading；禁止重复点击。 |
| **成功** | 关闭 Modal；消息区空；列表仍保留该条（§6）。 |
| **失败** | Modal 可关或保留错误信息；数据不变。 |

### 4.5 切换会话时的串会话防护（交互层）

- 切换瞬间：主消息区**不得**同时显示两个会话的混合内容。
- 若新会话消息请求较慢：显示加载态，**不**沿用旧会话消息。

## 5. 交互细则

### 5.1 新建入口位置

- **桌面**：历史区**顶部**主按钮「新建会话」（主色填充），次要操作（清空）放次要按钮或列表下方，避免与新建并列误点。
- **移动**：**顶栏左侧或右侧**图标「菜单/历史」打开 Drawer；Drawer **顶部**或 **主对话区顶部工具条** 放置「新建会话」。
- **可发现性**：新建在首屏无需滚动即可见（侧栏顶部或移动顶栏）。

### 5.2 当前选中会话高亮

- 列表项：**背景色区分**（如浅灰底）+ **左侧 3～4px 主色条**（可选）+ **字重 medium**。
- **键盘**：支持 ↑↓ 在列表中移动焦点（可选增强），Enter 选中。

### 5.3 清空二次确认（文案建议）

- **标题**：清空当前对话？
- **正文**：将删除**本对话中的全部消息**，对话仍保留在侧边列表中，之后可继续发送新消息。此操作不可撤销。
- **按钮**：取消（主焦点安全侧） / 清空（危险态，可用 `danger`）
- **禁用双击**：确认按钮在请求中 loading。

### 5.4 切换会话时输入草稿的处理（已定策略）

**策略：按会话 ID 隔离保存输入草稿（客户端）**

| 要点 | 说明 |
| --- | --- |
| **行为** | 输入框内容以 `conversationId` 为键保存在内存（或 `sessionStorage` 若需刷新保留，本迭代可仅用内存）。切换会话时，输入区**切换为**目标会话的草稿；若无草稿则空。 |
| **理由** | 符合「多会话并行准备不同内容」的心智；避免误将 A 会话文字发到 B；满足 PRD「切换后展示与当前会话一致」且不串会话；比「一律丢弃」更少挫败感；比「全局单草稿」更安全。 |
| **清空消息后** | 对该会话**同时清空其草稿**，与「在同一条目下重新开始」一致（与 PRD 3.4 意图对齐）。 |
| **边界** | 若会话极多且用持久化草稿，需注意存储上限（可仅最近 N 个会话或仅内存）。 |

## 6. 会话列表项字段建议

### 6.1 标题规则（收敛 O1）

| 规则 | 说明 |
| --- | --- |
| **默认** | 新创建且**尚无用户消息**时：显示 **「新对话」**。 |
| **有首条用户消息后** | 使用 **首条用户消息** 的前 **24～32 个字符**（中英文混合按显示宽度可再调），超出用 **「…」** 截断。 |
| **已清空** | 清空后若无任何消息：标题回退为 **「新对话」**（或 **「新对话（已清空）」** 若需更强区分，二选一由视觉权重定）。 |
| **未来扩展** | 用户可编辑标题（Out of 本迭代可不在 UI 实现）。 |

### 6.2 时间格式

| 场景 | 格式建议 |
| --- | --- |
| **当天** | 相对或绝对均可：如 `14:32` 或「今天 14:32」 |
| **昨天** | `昨天 14:32` |
| **本周内** | `周三 14:32`（locale 随站点语言） |
| **更早** | `YYYY-MM-DD` 或 `M月D日` |

列表排序：**按最后更新时间倒序**（与「最近在上」一致）。

### 6.3 辅助信息（可选）

- **预览副行**：最后一轮用户或助手消息前若干字（实现成本允许时）；否则可仅标题 + 时间。
- **空会话 / 已清空**：副行可为 **「暂无消息」** 或 **「内容已清空」**，避免用户误以为记录消失。

## 7. 可访问性要点

| 项 | 建议 |
| --- | --- |
| **焦点顺序** | 页面：主 landmark → 历史列表（可 Tab 进入第一条）→ 新建按钮 → 当前选中项 → 主区消息区（只读流）→ 输入区 → 发送。打开 Drawer/Modal 时焦点**移入**首可聚焦控件。 |
| **按钮语义** | 「新建会话」「打开历史」「清空当前对话」「发送」均用可辨识文案；图标按钮带 `aria-label`。 |
| **列表** | 会话列表容器 `role="navigation"` 或带 `aria-label="对话历史"`；当前选中项 `aria-current="true"`。 |
| **对话框陷阱** | 清空确认使用 Modal：`focus trap`、**ESC 关闭**、关闭后焦点回到触发「清空」的按钮。 |
| **加载** | 区域加载用 `aria-busy` 或 live region 简要声明（避免过度播报）。 |
